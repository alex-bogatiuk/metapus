# Развертывание Cloud SaaS (Версионирование)

> **TL;DR:** В облачном режиме (SaaS) мы можем крутить в одном кластере несколько разных версий сервера (например, v1.3 и v1.4) одновременно. Nginx автоматически маршрутизирует трафик к нужному серверу на основании версии, закрепленной за тенантом.

> **Тип:** How-To
> **Аудитория:** DevOps

---

## 1. Архитектура маршрутизации

Проблема: У нас 100 клиентов (тенантов). Мы хотим обновить 5 из них (Canary Release) до новой версии, а остальных оставить на старой.

**Решение:**
1. Запрос приходит в Nginx.
2. Nginx берет `X-Tenant-ID` и делает внутренний запрос (`auth_request`) в микросервис Router (специальный экземпляр Metapus).
3. Router смотрит в таблицу `tenants` мета-базы и возвращает ответ: `X-Version-Group: v1.4.0`.
4. Nginx через директиву `map` перенаправляет запрос на Docker-контейнер, у которого `VERSION_GROUP=v1.4.0`.

## 2. Как обновить тенантов

1. Соберите новый Docker образ (например, `metapus:v1.4.0`).
2. Добавьте его в `docker-compose.cloud.yml` как новый сервис с переменной `VERSION_GROUP=v1.4.0`.
3. Добавьте upstream в конфиг Nginx.
4. Выполните миграции БД для тенанта: `go run cmd/tenant/main.go migrate --id <UUID>`.
5. Переключите тенанта на новую версию: `go run cmd/tenant/main.go promote --id <UUID> --to v1.4.0`.

Ни один другой тенант не пострадает, так как их трафик продолжает идти на контейнер `v1.3.0`.

## 3. Откат (Rollback)

Если в `v1.4.0` обнаружена критическая ошибка, вернуть тенанта на старую версию — это одна команда (она просто меняет строку в базе метаданных):
```bash
go run cmd/tenant/main.go promote --id <UUID> --to v1.3.0
```
> [!WARNING]
> Откат кода возможен только в том случае, если новые SQL-миграции, примененные к БД тенанта в `v1.4.0`, обратно совместимы с кодом `v1.3.0` (например, добавлены новые колонки, а не удалены/переименованы старые).
