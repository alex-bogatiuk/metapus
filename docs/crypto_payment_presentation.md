# Шпаргалка: Архитектура криптопроцессинга Metapus

Эта шпаргалка предназначена для презентации техническому директору. Она описывает полный жизненный цикл криптоплатежа в нашей системе.

## 1. Как система определяет, оплачен ли инвойс?

Статус инвойса (модель `CryptoInvoice`) определяется строгим конечным автоматом (FSM), который сравнивает `ExpectedAmount` (сколько запросили) и `ReceivedAmount` (сколько реально поступило на кошелек).

*   **`created`**: Начальный статус. Инвойс ждёт оплаты, кошелек из пула заблокирован (Leased) под этот инвойс.
*   **`partially_paid`**: `ReceivedAmount` > 0, но меньше `ExpectedAmount`.
*   **`paid`**: `ReceivedAmount` точно равен `ExpectedAmount`.
*   **`overpaid`**: `ReceivedAmount` > `ExpectedAmount`. Система имеет `SweepConfigResolver`, который проверяет "толерантность" к переплате. Если переплата крошечная (dust), статус ставится `paid`. Если существенная — `overpaid`.
*   **`confirmed`**: Транзакция в блокчейне набрала нужное количество подтверждений (Required Confirmations). В этот момент инвойс окончательно закрывается, кошелек освобождается (или уходит на Sweep), а в системе создаются бухгалтерские проводки (Posting) по регистрам балансов.

> [!NOTE]
> Логика расчета скрыта в чистой функции `updateInvoiceAmount` в `EventProcessor`. Сумма может собираться из нескольких транзакций (например, клиент отправил двумя частями).

## 2. Как мы сканируем блокчейн?

За сканирование отвечает фоновый **Worker** (`cmd/worker`), который работает в мульти-тенантном режиме.

1.  **Запуск воркера**: Для каждого активного тенанта запускается свой экземпляр `CryptoProcessor`.
2.  **Загрузка сетей**: Процессор смотрит в БД (`cat_blockchain_networks`), какие сети активны (например, TRON).
3.  **Chain Watchers**: Запускаются отдельные горутины-вотчеры для каждой сети (например, `tron.Watcher`, использующий TronGrid API).
4.  **Мониторинг пула**: Вотчер загружает из БД все наши пуловые адреса кошельков и слушает новые блоки.
5.  **Генерация событий**: Когда в блоке находится транзакция на наш адрес, вотчер отправляет нормализованный `BlockchainEvent` (TxHash, Amount, Confirmations, NetworkID) в общий канал.
6.  **EventProcessor**: Центральный процессор событий берет транзакцию, находит кошелек по адресу, по кошельку находит активный инвойс и обновляет его статус.

## 3. Как информация попадает в виджет оплаты?

Ссылка вида `http://localhost:8080/pay/019e...` обслуживается напрямую Go-бэкендом без необходимости поднимать отдельный Next.js сервер для этой страницы.

1.  **Рендеринг**: При переходе по ссылке роутер отдает встроенный HTML-шаблон (embedded HTML `static/payment.html`).
2.  **Первичная загрузка**: JS на странице делает запрос к публичному API `GET /api/v1/pay/:invoiceId`, который возвращает данные для отображения (адрес кошелька, сумму, QR-код в base64, токен).
3.  **Real-time обновления**: Страница использует **Polling**. JS-клиент раз в несколько секунд дергает легковесный эндпоинт `GET /api/v1/pay/:invoiceId/status`.
4.  Этот эндпоинт делает один быстрый `SELECT` в БД, доставая текущий `status`, `received_amount` и агрегируя `MAX(confirmations)` из таблицы платежей (`doc_crypto_payments`).

> [!TIP]
> Мы выбрали Polling (а не WebSockets), так как страница оплаты — публичная, открывается тысячами пользователей с мобильных устройств (плохой интернет). Polling дешевле по ресурсам для серверов, отказоустойчив при обрывах соединения и легко кэшируется.

## 4. Каверзные вопросы от CTO (и как на них отвечать)

### Q: Что если клиент отправит 1000 транзакций по 0.000001 USDT (Dust Attack), чтобы завалить нам БД?
**A:** В `EventProcessor` реализована защита от пылевых атак. Транзакции с суммой ниже `DustThreshold` (по умолчанию 1000 minor units) просто отбрасываются до записи в БД.

### Q: Что если сканер блокчейна перезапустится и прочитает один и тот же блок дважды?
**A:** У нас строгая **идемпотентность**. Перед созданием сущности `CryptoPayment` система делает поиск по `TxHash`. Если транзакция уже есть в БД, система не создает дубль, а лишь обновляет количество подтверждений (`Confirmations`) у существующего платежа.

### Q: Клиент оплатил, но сеть зависла (нет подтверждений). Инвойс протухнет?
**A:** Нет. Как только мы видим транзакцию в мемпуле или первом блоке (1 confirmation), статус инвойса меняется на `paid` или `partially_paid`. Фоновый процесс `runExpirationLoop`, который проверяет TTL инвойсов, трогает **только** инвойсы в статусе `created`.

### Q: Как архитектура обеспечивает изоляцию данных (Multi-tenancy), если worker один?
**A:** `MultiTenantWorker` динамически получает список активных тенантов из мета-БД. Для каждого тенанта создается **изолированный контекст** с собственным пулом подключений `pgxpool` (через `TenantManager`), своим `CryptoProcessor` и своим Outbox. Данные разных мерчантов физически не могут пересечься на уровне запросов.

### Q: Каким запросом (M2M) мерчант создаёт инвойс из своей системы?
**A:** Мерчанты используют отдельный публичный API `/merchant/v1/invoices`, который защищен через `X-Api-Key`. Он спроектирован так, чтобы не требовать JWT токенов и скрывать внутренние ID системы.

**Пример запроса (Создание инвойса на 10.5 USDT):**
```http
POST /merchant/v1/invoices HTTP/1.1
Host: api.metapus.io
X-Api-Key: mt_live_xxxxx
X-Tenant-ID: 5cfe45cb-9035-4e57-93e2-127e960370b8
Content-Type: application/json

{
  "amount": 10500000, 
  "currency": "USDT_TRC20",
  "orderId": "order_777_xyz",
  "description": "Оплата заказа #777",
  "ttlMinutes": 60,
  "callbackUrl": "https://merchant-shop.com/webhook",
  "customerEmail": "client@example.com"
}
```

> Обрати внимание: сумма передается в **minor units** (копейках/сатоши). Для USDT с 6 знаками после запятой `10.5 USDT = 10500000`.

**Пример ответа:**
```json
{
  "id": "019e06db-9d31-7819-bf89-ab69b5c5dbff",
  "status": "created",
  "amount": 10500000,
  "currency": "USDT_TRC20",
  "network": "TRON-SHASTA",
  "walletAddress": "TCkM2u2j41XfR8v2k3Y4Q8n2o...",
  "expiresAt": "2026-05-14T14:30:00Z",
  "orderId": "order_777_xyz",
  "description": "Оплата заказа #777",
  "createdAt": "2026-05-14T13:30:00Z"
}
```

Мерчант получает `id` и может перенаправить клиента на страницу оплаты: `https://pay.metapus.io/pay/{id}`.

### Q: Что происходит с деньгами после подтверждения?
**A:** Срабатывает Sweep Engine. Если настроен автоматический свип (перевод на холодный кошелек), то кошелек помечается как `SweepPending`, а отдельная джоба формирует транзакцию вывода (Withdrawal) для консолидации средств, оплачивая комиссию сети с gas-кошелька тенанта. Если порог свипа не достигнут (Threshold mode), деньги лежат на транзитном кошельке и ждут накопления.
