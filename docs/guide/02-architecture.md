# 02. Архитектура

> **TL;DR:** Metapus использует Clean Architecture со строгим правилом зависимостей: внутренние слои (Core, Domain) не знают о внешних (Infrastructure, Presentation). Единственный источник истины — Go-структуры (Code is Metadata).

> **Тип:** Concept
> **Аудитория:** Developer
> **Связанные:** [01-overview.md](01-overview.md), [03-project-structure.md](03-project-structure.md)

---

## 1. Слои приложения

Зависимости всегда направлены внутрь. Domain ничего не знает о HTTP-обработчиках или SQL-запросах.

```
┌─────────────────────────────────────────────────┐
│  cmd/*          — Composition Root (DI, запуск) │
├─────────────────────────────────────────────────┤
│  infrastructure — Адаптеры (HTTP, Postgres)      │
├─────────────────────────────────────────────────┤
│  domain         — Бизнес-логика (Сервисы)        │
├─────────────────────────────────────────────────┤
│  core           — Фундаментальные типы (Entity)  │
└─────────────────────────────────────────────────┘
```

## 2. Core (Ядро)
**Назначение:** Фундаментальные строительные блоки, не имеющие зависимостей от других слоёв.

- **BaseEntity**: Базовые типы `BaseCatalog`, `BaseDocument` с общими полями (ID, CreatedAt, Version).
- **AppError**: Структурированные ошибки системы по RFC 7807 (Code, Message, Details).
- **Трейты**: Интерфейсы типа `CurrencyAware` или типы данных `MinorUnits`.
- **Tenant Context**: Утилиты для работы с Multi-tenancy без привязки к СУБД.

## 3. Domain (Бизнес-логика)
**Назначение:** Бизнес-правила (use cases). Структурируется по Bounded Contexts.

Каждый контекст (например, `catalogs/counterparty`) содержит:
1. `model.go` — структура данных и её метод `Validate(ctx)`. Никаких обращений в БД при валидации.
2. `repo.go` — **интерфейс** репозитория (Dependency Inversion). Домен диктует, как с ним должны работать.
3. `service.go` — оркестровка. Открывает транзакцию (через абстрактный `TxManager`), вызывает хуки, обращается к репозиторию.

## 4. Infrastructure (Реализация)
**Назначение:** Адаптеры для внешнего мира.

- **HTTP (Gin)**: Роутинг, Middleware, DTO. Конвертирует HTTP-запросы в вызовы сервисов. `Handler` — тонкий слой, он не содержит бизнес-логики.
- **Postgres (pgx, squirrel)**: Реализация интерфейсов репозиториев из Domain. Строит SQL-запросы, выполняет маппинг. Использует `TxManager` для транзакций.

## 5. Composition Root (cmd)
**Назначение:** Сборка приложения.

Собирает зависимости вручную (без DI фреймворков), читает конфигурацию из ENV, поднимает сервер, настраивает graceful shutdown.

```path

```
