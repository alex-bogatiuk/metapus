# Правила разработки

> **TL;DR:** Строгие инварианты при написании кода. Нарушение этих правил классифицируется как архитектурная или логическая ошибка.

> **Тип:** Reference
> **Аудитория:** Developer
> **Связанные:** [02-architecture.md](../guide/02-architecture.md)

---

## 1. Архитектура и слои

- **Core не зависит ни от чего**. Зависит только от стандартной библиотеки Go.
- **Domain зависит только от Core**. Никаких импортов `github.com/gin-gonic/gin` или `github.com/jackc/pgx/v5` в слое бизнес-логики.
- **Нет `any`/`interface{}`** в бизнес-логике. Если нужен обобщённый тип — используйте Go Generics (`[T any]`).
- **Handler — это тонкий адаптер**. Запрещено писать `if` с бизнес-условиями в HTTP-хэндлерах.

## 2. База данных и транзакции

> [!IMPORTANT]
> База данных является единой точкой синхронизации. 

- **Database-per-Tenant**. Запрещено добавлять `tenant_id` в бизнес-таблицы (документы, справочники).
- **В `reg_*_movements` таблицах нет UPDATE**. Immutable Ledger. Старые движения удаляются, новые пишутся.
- **Никаких ORM**. Только сырой SQL (`pgx`) и query builder (`squirrel`).
- **Сортировка ресурсов (Resource Ordering)**. При использовании `SELECT FOR UPDATE` для массовых обновлений (например, списании остатков) ресурсы **обязаны** быть предварительно отсортированы по ID для предотвращения deadlock-ов (AB-BA).

## 3. Обработка ошибок

- **Единая точка выхода**. Все ошибки всплывают наверх. В хэндлере: `c.Error(err) + c.Abort()`. 
- Запрещено использовать `c.JSON(status, error)` в хэндлерах напрямую. Этим занимается `ErrorHandler` middleware.
- **Оборачивание (Wrapping)**. Все ошибки должны оборачиваться для трейсинга: `fmt.Errorf("action description: %w", err)`.
- Использовать только ошибки `apperror.AppError` с явными кодами.

## 4. Многопоточность (Concurrency)

- **Контекст обязателен**. Каждая запущенная `go func()` должна принимать `context.Context` и слушать `<-ctx.Done()`. Запуск горутин без остановки — утечка памяти (Goroutine leak).
- В worker pool использовать семафоры и `sync.WaitGroup` для Graceful Shutdown.
- Запрещено писать в ResponseWriter (SSE/JSON) из нескольких горутин одновременно. Сбор данных идёт в канал, а пишет одна главная горутина.

## 5. Валидация и вычисления

- Метод `Validate(ctx)` структуры не имеет права ходить в БД или сеть. Только чистые проверки в памяти (числа > 0, строки не пустые).
- Метод `GenerateMovements()` для документов должен быть абсолютно **детерминированным** (один и тот же вход всегда даёт один и тот же выход без сайд-эффектов).

## 6. Миграции (Ранний этап)

> [!WARNING]
> Пока платформа на раннем этапе (нет production), **запрещено создавать новые миграции для исправления старых**. 

- Если нужно добавить поле в `cat_counterparties`, редактируйте существующий файл `00013_cat_counterparties.sql`.
- Схема базы данных пересоздаётся с нуля командой `make reset-db`. 
- Все новые таблицы должны содержать CDC-колонки (`created_at`, `updated_at`, `version`) и `TIMESTAMPTZ` (не `TIMESTAMP`).
