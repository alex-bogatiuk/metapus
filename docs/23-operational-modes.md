# Операционные режимы работы Metapus

Фундаментальной архитектурой хранения данных в платформе является **строгая изоляция Database-per-Tenant** (одна база данных = один тенант).
Платформа **не поддерживает** режимы "Shared DB" (когда данные разных клиентов лежат в одних таблицах и разделяются отдельной колонкой `tenant_id`).

Однако с точки зрения развертывания и предоставления сервиса, бинарный файл сервера поддерживает **три операционных режима** (Operational Modes).

---

## 1. Cloud SaaS (Multi-Version Mode)

Основной облачный режим, в котором одновременно запускаются несколько экземпляров ядра (бинарника) **разных версий**.

- **Как активируется:** У инстанса сервера задается переменная окружения `VERSION_GROUP=v1.3.0`.
- **Маршрутизация:** Nginx/перевернутый прокси через `auth_request` опрашивает базу метаданных (`metapus_meta.tenants`) по заголовку `X-Tenant-ID` и перенаправляет запрос на конкретный `upstream` нужной версии, в зависимости от привязанной к тенанту `VERSION_GROUP`.
- **Особенности:**
  - Каждый инстанс (включая воркеры) обслуживает только своих тенантов.
  - Позволяет обновлять платформу поэтапно (канареечный релиз, A/B тестирование новых версий бизнес-логики).
  - Успешная валидация происходит за счет «шлюза версий» (Version Gate): если запрос к тенанту случайно попадет на инстанс не его версии, сервер вернет код `421 Misdirected Request`.
- **Подробнее:** [22-cloud-deployment.md](22-cloud-deployment.md)

---

## 2. Self-Hosted (Standalone Multi-Tenant Mode)

Классический On-Premise режим для крупных компаний-холдингов с несколькими юридическими лицами (дочерними компаниями), требующими раздельного учета.

- **Как активируется:** Переменная `VERSION_GROUP` не устанавливается (пустая строка).
- **Маршрутизация:** Все входящие запросы отправляются на единый кластер серверов текущей версии. Nginx не содержит логики проверки версий, он просто балансирует трафик.
- **Особенности:**
  - В таблице мета-базы `tenants` поле `version_group` также остается пустым.
  - Администратор может управлять списком компаний (тенантов) через раздел **«Настройки → Тенанты»** или через CLI.
  - Обновление платформы происходит одновременно для всех баз данных — сначала прогоняются миграции `go run cmd/tenant/main.go migrate` (для всех тенантов сразу), затем рестартуется кластер серверов.
  
---

## 3. Dedicated / Single Tenant Mode

Режим установки, в котором система используется в качестве стандартной монопольной ERP-программы исключительно для одной компании-заказчика.

- **Как активируется:** Отдельного флага в бинарнике нет, это скорее инфраструктурный сценарий (также без установленной `VERSION_GROUP`). Кластер поднимается для обслуживания единственной компании.
- **Маршрутизация:** В базе `metapus_meta` создается ровно **1 (одна)** запись тенанта.
- **Особенности:**
  - Для заказчика система выглядит как выделенный экземпляр (Dedicated Instance).
  - Заголовок `X-Tenant-ID` на стороне клиента (фронтенда) фиксируется на уровне билда (через `.env.local` фронтенд-приложения `NEXT_PUBLIC_TENANT_ID=...`) либо подставляется прокси-сервером автоматически при каждом запросе: `proxy_set_header X-Tenant-ID "уникальный-id";`.
  - При таком подходе пользователи даже не подозревают о multi-tenant природе бэкенда.

---

**Database-per-Tenant**

В Metapus изоляция всегда происходит на уровне выбора физического пула соединений к базе данных в Middleware:
```go
// 1. По X-Tenant-ID получаем UUID
// 2. Достаем пул соединений к нужной базе (e.g. pg_mt_acm_corp)
managedPool, _ := manager.GetPool(ctx, tenantID)
// 3. Кладем его в контекст: бизнес-логика ничего не знает о других БД
ctx = tenant.WithTxManager(ctx, txManager)
```
Поскольку SQL-запросы в системных слоях (Domain/HTTP/Repo) пишутся строго без WHERE-условий вроде `WHERE tenant_id = ?`, использовать единую базу (Shared BD) для нескольких тенантов технически невозможно (привело бы к утечке данных). В итоге параметр изоляции был убран из конфигурации для избежания путаницы у системных администраторов.
