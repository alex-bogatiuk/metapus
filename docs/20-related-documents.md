# Related Documents (Связанные документы)

Система «Связанные документы» предоставляет древовидное отображение структуры подчиненности (basis hierarchy) и автоматические карточки предварительного просмотра (Preview Cards) при наведении.

## Настройка превью-карточек (Zero-Config)

Отображение данных в превью-карточках работает на основе **метаданных**, которые собираются автоматически при регистрации сущности (документа).

### 1. Автоматический сбор полей
`metadata.Inspector` сканирует поля шапки документа (top-level fields) и ищет ссылки (References).
- Любое поле типа `id.ID`, ссылающееся на справочник (через реестр метаданных), автоматически попадает в список превью-полей.
- Системные поля (`id`, `organization_id`, `basis_id`, `created_by`, `updated_by` и т.д.) исключаются автоматически.

### 2. Как добавить поле в превью
Достаточно добавить поле-ссылку в структуру документа. Метка поля (Label) будет взята из тега `meta` или угадана по имени поля.

```go
type GoodsReceipt struct {
    entity.Document
    // Появится в карточке превью с лейблом "Поставщик"
    SupplierID id.ID `db:"supplier_id" json:"supplierId" meta:"label:Поставщик"`
    
    // Появится в карточке превью с лейблом "Склад"
    WarehouseID id.ID `db:"warehouse_id" json:"warehouseId" meta:"label:Склад"`
}
```

### 3. Как исключить поле из превью
Если поле-ссылка не должно отображаться в карточке (например, техническая ссылка), используйте тег `meta:"preview:false"`.

```go
type InternalDoc struct {
    entity.Document
    // Будет скрыто из карточки превью в связанных документах
    TechnicalRefID id.ID `db:"tech_id" json:"techId" meta:"preview:false"`
}
```

### 4. Техническая реализация
1. **Инспектор**: `internal/metadata/inspector.go` заполняет `EntityDef.PreviewFields`.
2. **Resolver**: `internal/infrastructure/storage/postgres/ref_resolver_repo.go` генерирует динамические `LEFT JOIN` для всех превью-полей. Это позволяет получить все наименования (имена контрагентов, складов и т.д.) за **один SQL-запрос** на каждый тип сущности в дереве, избегая проблемы N+1.
3. **Frontend**: `RelatedDocItem` содержит мапу `previewData` (Label -> Value). Компонент `DocumentPreviewCard` рендерит эту мапу динамически.

## Дерево подчиненности (Recursive Tree)

Дерево строится путем обхода ссылок `basis_type` и `basis_id`.

1. **Обход ВВЕРХ**: Система находит корень всей цепочки документов.
2. **Обход ВНИЗ**: От корня строится полное дерево через BFS (поиск всех документов, где `basis_id` указывает на текущий узел).
3. **Лимиты безопасности**: 
   - Максимальная глубина: 10
   - Максимальное число узлов: 100
   
Эти параметры настраиваются в `internal/infrastructure/storage/postgres/related_doc_repo.go`.

## Прочие ссылки (Flat Groups)

Ссылки, которые не являются «основанием» (basis), но указывают на текущий документ (например, произвольные FK из других таблиц), отображаются в разделе «Прочие ссылки» в виде плоских групп. Сбор этих ссылок осуществляется через `RefFinderRepo`.
