# Уведомления реального времени (Real-Time Notifications)

> **TL;DR:** Система использует WebSockets для push-уведомлений. Соединения строго изолированы по `tenant_id`. Воркеры отправляют уведомления пачками через `InternalNotificationAdapter`, а фронтенд обновляет UI без перезагрузки.

> **Тип:** Concept
> **Аудитория:** Developer
> **Связанные:** [automation-engine.md](automation-engine.md)

---

## 1. Архитектура доставки

Уведомления редко генерируются "жестким кодом". Вместо этого используется подсистема `Automation Engine` (Automation Rules).

1. **Automation Engine:** Правило автоматизации срабатывает, CEL-шаблон рендерит payload (JSON с текстом и ссылкой), а адаптер `InternalNotificationAdapter` готовит пачку уведомлений.
2. **Database:** Уведомления сохраняются в таблицу `sys_notifications` за 1 SQL запрос.
3. **WebSocket Hub:** Бэкенд (`GlobalHub`) отправляет сообщение по WebSocket **только тем пользователям**, которые сейчас в онлайне. Соединения в Hub хранятся в виде `map[tenantID][userID]`, что гарантирует строгую изоляцию тенантов.

## 2. Клиентская сторона (Frontend)

Фронтенд инкапсулирует логику в хук `useWebsocket`.

- **Аутентификация:** Так как WebSocket не поддерживает заголовок `Authorization`, JWT токен передается в URL параметре `?token=<token>`.
- **Оптимистичный UI:** При клике на уведомление в компоненте `NotificationBell`, отправляется запрос `markAsRead`, и локальный кеш сразу обновляется (счетчик колокольчика уменьшается), без необходимости перезапрашивать весь список с сервера.
- **Восстановление связи:** При обрыве соединения используется алгоритм Exponential Backoff (1с, 2с, 4с... max 30с) для переподключения.
