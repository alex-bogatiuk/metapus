# Движок проведения (Posting Engine)

> **TL;DR:** Документы не пишут в регистры напрямую. Они реализуют интерфейсы-источники движений (Source), а центральный Posting Engine координирует транзакцию, контроль остатков и массовое проведение (batch operations).

> **Тип:** Concept
> **Аудитория:** Developer
> **Связанные:** [transactions.md](transactions.md)

---

## 1. Паттерн «Посетитель» (Visitor Pattern)

Система проведения decoupled (развязана). Документ знает только о своих движениях, но не о том, как они записываются.

1. **Документ реализует Source-интерфейс**. Например: `StockMovementSource`.
2. Документ формирует массив движений (`[]entity.StockMovement`) на основе своих строк.
3. **Posting Engine** прогоняет набор `Visitor`-ов по документу. Если документ реализует `StockMovementSource`, `StockVisitor` собирает его движения и кладёт в общую "корзину" (`MovementSet`).

## 2. Фазы проведения документа

Проведение всегда выполняется в единой транзакции (REPEATABLE READ).

1. **Валидация (до транзакции):** `doc.CanPost(ctx)` (например, проверка заполненности складов).
2. **Reverse (Перепроведение):** Если документ был проведён ранее, старые движения полностью удаляются. 
3. **Сбор движений:** Visitor собирает новые движения из документа.
4. **Контроль остатков (Pessimistic Lock):** Для расходных движений. Движок сортирует товары по ID (Resource Ordering, защита от deadlock) и делает `SELECT FOR UPDATE`.
5. **Запись движений (Batch Insert):** Новые движения записываются в таблицы регистров сведений/накопления через `pgx.CopyFrom` (сверхбыстрая вставка).
6. **Триггеры БД:** Таблицы остатков (`reg_stock_balances`) обновляются автоматически через PostgreSQL триггеры `AFTER INSERT/DELETE`. Документ **никогда** не обновляет остатки руками.
7. **Фиксация статуса:** Документу ставится `Posted = true` и инкрементируется `PostedVersion`.

## 3. Immutable Ledger

Движения регистров **нельзя обновлять** (нет операции `UPDATE`).
Если документ изменился, его старые движения удаляются (и триггеры откатывают остатки), а затем записываются новые движения новой версии (и триггеры снова пересчитывают остатки).

## 4. Массовое проведение (Batch Posting)

Система поддерживает параллельное проведение тысяч документов с отображением прогресса на клиенте.

| Режим | Описание |
|-------|----------|
| **По массиву ID** | Пользователь выделил конкретные галочки. Передаётся массив до 500 ID. |
| **Virtual Select All** | Пользователь нажал "Выбрать всё по фильтру". Сервер сам разрешает ID (до 100 000 шт) на основе фильтров. |

### Особенности параллельной обработки:
- **Worker Pool:** Количество параллельных горутин (`batchConcurrency`) настраивается в `sys_settings` и ограничено пулом коннектов к БД.
- **Изоляция:** Каждый документ проводится в своей транзакции. Падение одного не откатывает остальные.
- **SSE Стриминг:** Если клиент прислал `Accept: text/event-stream`, сервер открывает SSE-канал и шлёт события прогресса (`{"processed": 50, "total": 2000}`) без буферизации.
