# Управление транзакциями

> **TL;DR:** Все транзакции управляются через `TxManager`, который инжектируется через `context.Context`. Система поддерживает оптимистичные блокировки по умолчанию, пессимистичные (FOR UPDATE) для остатков и вложенные транзакции (savepoints).

> **Тип:** Concept
> **Аудитория:** Developer
> **Связанные:** [multi-tenancy.md](multi-tenancy.md)

---

## 1. Абстракция `tx.Manager`

Слой Domain никогда не вызывает `pgx.Begin()`. Он зависит от интерфейса:

```go
type Manager interface {
    RunInTransaction(ctx context.Context, fn func(ctx context.Context) error) error
}
```

Это позволяет легко мокать транзакции в юнит-тестах и изолирует бизнес-логику от драйвера базы данных.

## 2. Optimistic Locking (Оптимистичные блокировки)

В Metapus это **стратегия по умолчанию** для всех CRUD операций (Справочники, Документы).

1. Каждая сущность имеет поле `Version int`.
2. При обновлении репозиторий генерирует SQL вида:
   `UPDATE ... SET ..., version = version + 1 WHERE id = $1 AND version = $2 RETURNING version`
3. Если количество обновлённых строк = 0, значит, запись была изменена кем-то другим.
4. Возвращается ошибка `apperror.NewConcurrentModificationError()`, которая конвертируется в **HTTP 409 Conflict**.

## 3. Pessimistic Locking (Пессимистичные блокировки)

Используются **только для конкурентных вычислений**, где конфликты очень вероятны. Главный пример — контроль остатков при проведении документа.

1. Документ делает `SELECT ... FOR UPDATE` по товарам на складе.
2. Эта блокировка запрещает другим транзакциям менять остатки этих же товаров до конца текущей транзакции (COMMIT/ROLLBACK).
3. **Resource Ordering (Сортировка ресурсов):** Чтобы избежать взаимоблокировок (Deadlocks) типа A-B / B-A, перед `SELECT FOR UPDATE` движок проведения **всегда сортирует ID товаров** по алфавиту.

## 4. Вложенные транзакции (Nested Transactions)

Что если мы вызываем `RunInTransaction` внутри уже открытой транзакции?

- **Reuse (По умолчанию):** Метод просто выполнит код в контексте текущей транзакции (без SQL команды `BEGIN`). Ошибка внутри вложенной функции приведёт к ROLLBACK всей внешней транзакции.
- **Savepoint (`UseSavepoint: true`):** Метод создаст точку сохранения `SAVEPOINT sp_...`. Ошибка внутри вложенной функции сделает `ROLLBACK TO SAVEPOINT`, отменив только локальные изменения, и позволит внешней транзакции продолжить работу. (Используется редко, так как savepoints негативно влияют на производительность PostgreSQL).

## 5. GetQuerier Паттерн

Репозитории не знают, работают ли они внутри транзакции или вне её. 
Метод `GetQuerier(ctx)` возвращает либо `pgx.Tx` (если в контексте есть транзакция), либо `pgxpool.Pool` (если её нет). 

```go
func (r *Repo) Create(ctx context.Context, e *Entity) error {
    querier := r.getTxManager(ctx).GetQuerier(ctx)
    _, err := querier.Exec(ctx, "INSERT ...")
    return err
}
```
Это делает методы репозитория универсальными.
