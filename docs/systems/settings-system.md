# Система настроек (Settings System)

> **TL;DR:** Настройки хранятся на трёх уровнях (Система, Организация, Пользователь). UI строится динамически из реестра метаданных (Metadata-driven), поэтому добавление новой настройки не требует создания React-компонентов.

> **Тип:** Concept
> **Аудитория:** Developer

---

## 1. Трехуровневая модель

При добавлении нового параметра выберите правильный уровень:

| Уровень | Таблица | Scope | Описание / Пример |
|---------|---------|-------|-------------------|
| **Системный** | `sys_settings` | Тенант | Глобальные настройки тенанта (Правила нумерации документов, лимиты параллелизма). Доступно только администраторам. |
| **Организационный** | `cat_organizations` | Юрлицо | Настройки, привязанные к конкретной компании (Учетная политика, НДС, базовая валюта). |
| **Пользовательский** | `user_preferences` | Юзер | Личные UI/UX предпочтения (Темная тема, формат дат, компактный режим списков). |

## 2. Системные настройки (sys_settings)

Таблица `sys_settings` имеет структуру с ограничением `CHECK (singleton = TRUE)`. В ней **всегда ровно одна строка**.

Каждая логическая секция настроек (Общие, Склад, Закупки) — это отдельная JSONB колонка (`numbering`, `performance`, `warehouse`). Это позволяет обновлять секции независимо друг от друга, не вызывая конфликтов (Optimistic Locking).

## 3. Metadata-driven UI

Фронтенд не имеет жестко запрограммированных форм для настроек. Он обращается к `/api/v1/meta/settings/sections`, получает структуру (какие вкладки существуют, какие там поля, типы полей, `label`, `description`) и рендерит `AutoForm` внутри `SettingsSection`.

**Как добавить настройку в код:**
1. Добавьте поле в Go-структуру нужной секции (например, `WarehouseSettings`).
2. Добавьте мета-теги: `meta:"label:Метод списания;widget:select;options:FIFO,LIFO"`.
3. Добавьте поле в валидацию (если нужно).
4. Все готово! UI сгенерируется автоматически.
