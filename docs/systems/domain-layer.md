# Слой Domain (Бизнес-логика)

> **TL;DR:** Доменный слой оркеструет бизнес-правила. Он изолирован от веб-фреймворков и SQL. Включает в себя Справочники, Документы, Регистры и систему Хуков.

> **Тип:** Concept
> **Аудитория:** Developer
> **Связанные:** [architecture.md](../guide/02-architecture.md), [crud-pipeline.md](crud-pipeline.md)

---

## 1. Структура (Bounded Contexts)

Код сгруппирован по бизнес-областям (Bounded Contexts), а не по типу файлов.
Пример контекста контрагентов (`internal/domain/catalogs/counterparty/`):
- `model.go` — структура данных и метод `Validate(ctx)`.
- `repo.go` — интерфейс репозитория (реализация будет в Infra).
- `service.go` — бизнес-логика.
- `hooks.go` — бизнес-события (хуки).

## 2. Модели и Валидация

Метод `Validate(ctx)` проверяет **инварианты сущности в памяти**. 
- **Разрешено:** проверка `Amount > 0`, регулярные выражения (формат email).
- **Запрещено:** запросы к БД (например, проверка уникальности ИНН) или к внешним API. Для этого используются **Хуки**.

## 3. Документы: Генерация движений

Документ является источником движений (Source) для движка проведения. Функция `GenerateMovements()` должна быть **детерминированной** (без side-effects):

```go
func (i *Invoice) GenerateMovements(version int) []StockMovement {
    // Чистая математика: маппинг строк (Items) в движения регистра (StockMovement)
    // Возвращает массив движений. Не обращается к БД!
}
```

## 4. Жизненный цикл (State)

У документов есть чёткая конечная машина состояний (State Machine):
- `Draft` (Черновик)
- `Posted` (Проведён)
- `MarkedForDeletion` (Помечен на удаление)

Вместо проверок `if doc.Posted`, используется паттерн State: `doc.State().CanModify()`, который инкапсулирует правила изменения документа.

## 5. Система Хуков (Hooks)

Все побочные эффекты (сайд-эффекты) выполняются через систему событий `HookRegistry`. 
Есть 6 точек расширения:
- `BeforeCreate` / `AfterCreate`
- `BeforeUpdate` / `AfterUpdate`
- `BeforeDelete` / `AfterDelete`

> [!NOTE]
> **Before-хуки** (например, генерация автономера, проверка ИНН в базе) выполняются ДО транзакции. Если они вернут ошибку — сохранение прервётся. **After-хуки** выполняются ПОСЛЕ коммита, ошибки в них только логируются.
