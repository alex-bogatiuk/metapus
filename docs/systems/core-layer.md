# Слой Core (Ядро)

> **TL;DR:** Фундаментальные типы (Entity, Error, UUIDv7, Tenant), от которых зависят все остальные слои. Ядро ничего не знает о бизнес-логике, базе данных или HTTP.

> **Тип:** Concept
> **Аудитория:** Developer
> **Связанные:** [architecture.md](../guide/02-architecture.md)

---

## 1. Базовые сущности (Entities)

Все бизнес-объекты строятся путём композиции базовых структур.

### BaseEntity
Содержит поля, общие для абсолютно всех сущностей:
- `ID` — UUIDv7 (уникальный и сортируемый по времени).
- `DeletionMark` — флаг мягкого удаления (soft delete).
- `Version` — счётчик для оптимистичных блокировок (Optimistic Locking).
- `Attributes` — `map[string]any` (хранится как JSONB) для расширения сущности произвольными полями.

### BaseCatalog (Справочник)
Расширяет `BaseEntity`, добавляя:
- `Code` — уникальный строковый код справочника.
- `Name` — наименование.
- Поддержка иерархии (через `ParentID` и флаг `IsFolder`), которая регулируется метаданными.

### BaseDocument (Документ)
Документы фиксируют хозяйственные операции, поэтому они добавляют аудит-поля:
- `Number` и `Date` — номер и дата документа.
- `Posted` и `PostedVersion` — статус проведения.
- `OrganizationID` — к какой организации относится документ.
- Поля аудита: `CreatedAt`, `UpdatedAt`, `CreatedBy`, `UpdatedBy`.

## 2. Трейты (Mixins) и TypedRef

В Go нет наследования, поэтому используется встраивание (embedding).
- **Трейты:** Если документу нужна валюта, в него встраивается структура `CurrencyAware` (добавляет `CurrencyID`).
- **TypedRef:** Специальный составной тип (`RefType` + `RefID`) для полиморфных ссылок. Аналог "ОписаниеТипов" из 1С. Позволяет одному полю ссылаться на любые другие сущности.

## 3. Ошибки (AppError)

Система использует строго типизированные ошибки по стандарту RFC 7807:

```json
{
  "code": "INSUFFICIENT_STOCK",
  "message": "Недостаточно товара на складе",
  "details": {"product": "...", "requested": 10, "available": 5}
}
```

Конструкторы лежат в `apperror.New...`. В хэндлере ошибка передаётся в `c.Error(err)`.

## 4. Контекст Тенанта (Tenant Context)

> [!IMPORTANT]
> Ядро предоставляет методы `tenant.WithTxManager` и `tenant.MustGetTxManager` для работы с контекстом `context.Context`. Именно благодаря этому слой Domain может управлять транзакциями без прямой зависимости от пакета `pgx` или `sql`.
