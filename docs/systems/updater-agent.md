# Обновления (Updater Agent)

> **TL;DR:** Updater Agent — это сервис-"sidecar" для режима Single Tenant, который обновляет систему из UI с почти нулевым простоем (Near-Zero Downtime) используя Docker Network Alias Swap.

> **Тип:** Concept
> **Аудитория:** DevOps
> **Связанные:** [operational-modes.md](operational-modes.md)

---

## 1. Архитектура Blue-Green обновления

В режиме Single Tenant администратор может обновить систему прямо из интерфейса. Бэкенд сам себя обновить не может, поэтому используется сторонний сервис (`cmd/updater`), висящий на порту `:9090`, у которого есть доступ к `/var/run/docker.sock`.

**Как происходит "Connect-First Alias Swap":**
1. **Pull:** Агент скачивает новый образ из GHCR.
2. **Start:** Поднимает новый контейнер, но без привязки портов хоста.
3. **Health Check:** Ждет, пока новый контейнер ответит `200 OK` на `/health`.
4. **Connect:** Привязывает Docker Alias `metapus-app` к новому контейнеру (теперь трафик от Nginx идет на оба контейнера).
5. **Disconnect:** Отключает Alias от старого контейнера (весь трафик плавно переходит на новый).
6. **Migrate:** Вызывает внутренний эндпоинт нового контейнера для запуска миграций БД.
7. **Cleanup:** Останавливает старый контейнер.

## 2. Crash Recovery и Rollback

Агент ведет журнал состояний (WAL) в `state.json`. Если сервер выключат по питанию посреди обновления, агент при старте считает файл:
- Если упали на фазе скачивания или старта нового контейнера — он просто удалит "мусор" и сбросится в IDLE.
- Если упали на фазе переключения трафика — он восстановит Alias для старого контейнера.
- Если упали во время миграции БД — процесс переходит в статус FAILED и требует ручного отката (через API `/updater/rollback`).
