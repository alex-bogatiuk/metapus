# Подсистема Автоматизации (Automation Engine)

> **TL;DR:** Механизм реакции на бизнес-события или расписание (CRON). Двухфазное выполнение (Evaluate → Deliver), трёхуровневая адресация (Account → Channel → Subscriber) и AES-256 шифрование ключей.

> **Тип:** Concept
> **Аудитория:** Developer, Admin
> **Связанные:** [core-layer.md](core-layer.md)

---

## 1. Трёхуровневая модель доставки

Для безопасного управления доступом используется абстракция из трёх слоёв:

1. **Account (Аккаунт):** Учётные данные (например, Bot Token для Telegram или SMTP логин). Хранит зашифрованный секрет.
2. **Channel (Канал):** Конкретный адрес доставки (например, ID чата в Telegram), привязанный к Аккаунту.
3. **Subscriber (Подписчик):** Связка между Правилом автоматизации и Каналом.

## 2. Двухфазный движок

Бизнес-логика (например, проведение документа) генерирует событие в таблицу `sys_outbox`. Фоновый воркер (Outbox Consumer) подхватывает событие и запускает Automation Engine.

Процесс разбит на две фазы для отказоустойчивости:

### Фаза 1: Evaluate (CPU-bound)
Сбор правил и вычисление.
- Загрузка активных правил для события.
- Проверка Cooldown (защита от спама).
- **Выполнение CEL-условий** (например: `doc.totalAmount > 100000`).
- Рендеринг текста сообщения через Go Templates.
Возвращает список готовых к отправке заданий.

### Фаза 2: Deliver (I/O-bound)
Доставка и запись истории.
- Вызов адаптеров (`TelegramAdapter`, `EmailAdapter`, `WebhookAdapter`).
- Если адаптер падает — транзакция Outbox **не откатывается** (Immutable Ledger). Ошибка пишется в таблицу истории `sys_automation_history`, после чего срабатывает механизм повторов (Retries).

## 3. Безопасность: Шифрование Учёток

Все токены и пароли в таблице Accounts шифруются по стандарту **AES-256-GCM**.
- Ключ берётся из переменной окружения `AUTOMATION_ENCRYPTION_KEY` (строго 32 байта).
- В БД ложится: `nonce (12 байт) + ciphertext + auth_tag (16 байт)`.
- Учётные данные **никогда не возвращаются** по API (даже админам). Если нужно изменить пароль — отправляется новый Plaintext через `PATCH`.

## 4. Цепные реакции (Chain)

Движок поддерживает `reaction_type = chain`. Вместо отправки уведомления наружу, правило может генерировать **новое внутреннее событие** (публикация в Outbox), запуская каскад автоматизаций.
