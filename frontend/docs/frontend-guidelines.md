# Metapus Frontend Guidelines

Этот документ описывает принципы разработки frontend-части проекта Metapus.
Он адаптирует архитектурные правила backend (Clean Architecture, Metadata-driven) к экосистеме Next.js / React.

Всё что касается фронтэнда находится в папке - \frontend

---

## 1. Фундаментальные принципы

### 1.1. Code is Metadata (Frontend Edition)
UI строится на основе строгих контрактов (TypeScript interfaces), которые должны соответствовать структурам backend.
*   **Правило**: Не используйте `any` или слаботипизированные объекты.
*   **Правило**: Если backend возвращает DTO, frontend должен иметь идентичный интерфейс.

### 1.2. Explicit over Implicit
Избегайте "магии" в передаче данных и состояния.
*   **Правило**: Явно передавайте props. Избегайте глубокого prop drilling, но и не злоупотребляйте Context API для всего подряд.
*   **Правило**: Используйте URL как источник истины для состояния фильтров, пагинации и сортировки (URL-driven state).

### 1.3. Component isolation & UI Kit
Компоненты должны быть максимально изолированы и переиспользуемы.
*   **Правило**: UI-компоненты (`components/ui`) не должны знать о бизнес-логике.
*   **Правило (Strict)**: Весь UI/UX строится **исключительно** на базе библиотеки [shadcn/ui](https://ui.shadcn.com). Запрещено писать свои базовые компоненты (Button, Input, Modal и т.д.) с нуля, если аналог существует в shadcn или это не указано явно.

## 2. Архитектура (Vertical Slices + Feature-Sliced Design)

Мы используем гибридный подход, вдохновленный FSD и Vertical Slices, адаптированный под Next.js App Router.

### Структура папок

```
frontend/
├── app/                  # Pages & Routing (Composition Root)
│   ├── (auth)/           # Route groups
│   ├── catalogs/         # Slice: Справочники
│   │   ├── [type]/       # Dynamic catalog pages
│   │   └── page.tsx      # Entry point
│   ├── documents/        # Slice: Документы
│   └── layout.tsx        # Root layout
├── components/
│   ├── ui/               # Low-level UI kit (Shadcn/ui) - DUMB
│   ├── shared/           # Shared business components (e.g., UserNav)
│   └── [feature]/        # Feature-specific components (optional)
├── lib/                  # Utilities, helpers
├── hooks/                # Global hooks
└── types/                # Global types (API DTOs, Enums)
```

### Разделение ответственности

1.  **Page Components (`/app/**/page.tsx`)**:
    *   Являются **Server Components** (по умолчанию).
    *   Отвечают за получение данных (fetch).
    *   Передают данные в клиентские компоненты.
    *   Служат "точкой сборки" для фичи.

2.  **Smart Components (Containers)**:
    *   Содержат бизнес-логику и состояние (useState, useQuery).
    *   Могут делать запросы к API (если это Client Component).
    *   Собирают Dumb-компоненты.

3.  **Dumb Components (UI)**:
    *   Только отображение (`components/ui`).
    *   Получают данные и колбэки через props.
    *   Не имеют зависимости от внешнего мира (API, Context).

## 3. State Management

### 3.1. Server State (React Query / SWR / fetch)
Для данных, приходящих с сервера, используйте механизмы кэширования и ревалидации.
*   В Server Components: прямой `fetch`.
*   В Client Components: `useQuery` (если нужно) или передача initial data с сервера.

### 3.2. URL State (Single Source of Truth)
Состояние списка (фильтры, сортировка, страница, поиск) **должно** храниться в URL search params.
*   Это обеспечивает workability кнопок "Назад/Вперед" и sharing ссылок.
*   Используйте хуки для синхронизации URL и состояния (например, `nuqs` или кастомные).

### 3.3. Local State
Используйте `useState` / `useReducer` только для UI-состояния (открыт/закрыт диалог, значение инпута до сабмита).

### 3.4. Global State (Zustand / Context)
Используйте крайне осторожно. Только для действительно глобальных вещей:
*   Тема (Dark/Light)
*   Данные текущего пользователя/сессии
*   Настройки приложения

## 4. Код и Стиль

### 4.1. Naming Conventions
*   **Компоненты**: `PascalCase` (e.g., `Button.tsx`, `UserProfile.tsx`).
*   **Хуки**: `camelCase` с префиксом `use` (e.g., `useAuth.ts`).
*   **Утилиты/Функции**: `camelCase` (e.g., `formatDate.ts`).
*   **Папки в `app`**: `kebab-case` (next.js convention).

### 4.2. TypeScript
*   Всегда описывайте `interface` для Props.
*   Избегайте `FC<Props>`, предпочитайте явное деструктурирование:
    ```tsx
    // Good
    export function MyComponent({ title, isActive }: MyComponentProps) { ... }
    ```

### 4.3. Стилизация (Tailwind CSS)
*   Используйте `cn()` утилиту (clsx + tailwind-merge) для условных классов.
*   Придерживайтесь токенов дизайн-системы (цвета `background`, `foreground`, `primary` и т.д.), не хардкодьте hex-цвета.

### 4.4. Обработка ошибок
*   Используйте `error.tsx` для перехвата ошибок рендеринга (Error Boundary).
*   Для ошибок API показывайте `toast` (Sonner/Toast).
*   Валидируйте формы с помощью **Zod** + **React Hook Form**.

### 4.5. UI Kit & MCP Workflow (Shadcn/ui)
Мы используем **shadcn/ui** как основу дизайн-системы.
*   **Использование MCP**: При генерации кода для UI-компонентов или создании новых страниц Агент **обязан** использовать MCP (Model Context Protocol) для `shadcn/ui` (или доступ к документации), чтобы получить эталонный код компонента.
    *   ❌ **Don't**: Не генерируйте код кнопок или инпутов "из головы" или на основе старых знаний.
    *   ✅ **Do**: Сделайте запрос через MCP к https://ui.shadcn.com, получите актуальный код компонента и инструкцию по установке, и используйте их.
*   **Installation**: Новые компоненты копируются в папку `components/ui` в строгом соответствии с документацией.

## 5. Header Tabs (Browser-like Tab System)

Приложение использует кастомную систему вкладок в хедере (`SiteHeader`), стилизованных под "папки" (folder tabs). Это ключевой UX-паттерн проекта.

### 5.1. Архитектура

*   **Store**: `stores/useTabsStore.ts` (Zustand). Хранит массив `tabs`, `activeTabId` и экшены `openTab`, `setActiveTab`, `closeTab`, `setTabDirty`.
*   **Компоненты**:
    *   `components/layout/site-header.tsx` -- рендерит полосу вкладок.
    *   `components/layout/app-shell.tsx` -- открывает новую вкладку при навигации (через `usePathname`).
    *   `components/layout/app-sidebar.tsx` -- открывает вкладку при клике по пункту меню.

### 5.2. Визуальные правила (Folder Tab Effect)

Реализация вкладок имеет **строгие** визуальные правила:

**Header (контейнер)**:
*   Фон: `bg-muted/40`, высота: `h-12`, выравнивание вкладок по низу: `items-end`.
*   Нижняя граница реализована **не через `border-b`**, а через псевдоэлемент `after:`:
    ```
    after:pointer-events-none after:absolute after:inset-x-0 after:bottom-0 after:h-px after:bg-border
    ```
    Это критически важно -- `border-b` на `<header>` невозможно перекрыть дочерним элементом. Псевдоэлемент имеет стандартный z-index и может быть перекрыт вкладкой с `z-10`.

**Активная вкладка** (должна сливаться с контентом страницы):
*   `bg-background` (белый фон, совпадает с рабочей областью).
*   `border-x border-t border-border` -- рамка слева, справа и сверху.
*   `border-t-primary border-t-2` -- акцентная линия сверху.
*   `relative z-10 -mb-px` -- наложение: отрицательный margin-bottom на 1px вытягивает вкладку вниз, перекрывая линию хедера. `z-10` гарантирует, что вкладка выше псевдоэлемента.
*   **Нет `border-b`** -- нижняя граница отсутствует, белый фон плавно переходит в страницу.

**Неактивные вкладки** (утоплены на задний план):
*   `bg-muted/30` -- приглушённый фон.
*   `border-x border-t border-border` -- рамка со всех сторон (визуально за линией).
*   Нет `z-10` и `-mb-px` -- вкладка не перекрывает линию хедера.

**Separator**: Вертикальный разделитель (`<div className="h-6 w-px bg-border" />`) между `SidebarTrigger` и списком вкладок.

### 5.3. Dirty State (Несохранённые изменения)

*   **Хук `useTabDirty()`**: Вызывается **без аргументов** в формах. Он автоматически определяет ID вкладки через текущий `pathname`.
*   **Механизм отслеживания**: 
    - Основной контейнер формы (Smart Component) должен иметь обработчик `onChange={markDirty}`, который ловит всплывающие нативные события `change` от любых инпутов.
    - Для изменений, не вызывающих `change` (например, добавление/удаление строк в табличных частях), необходимо вызывать `markDirty()` явно.
*   **Жизненный цикл**: Состояние `isDirty` **не сбрасывается** при размонтировании (unmount) компонента. Это позволяет переключаться между вкладками, сохраняя индикатор изменений.
*   **Сброс (`markClean()`)**: Метод должен вызываться явно после успешного выполнения операций "Записать" или "Провести".
*   **Визуализация**: Индикатор в виде красной точки `●` (`text-destructive`) рядом с заголовком в соответствующей вкладке.
*   **Защита**: При попытке закрыть вкладку (нажать "X") с активным `isDirty`, `SiteHeader` перехватывает событие и показывает `AlertDialog` с подтверждением закрытия без сохранения.

### 5.4. Правила при реализации и изменении

*   **Запрещено**: использовать `border-b` на компоненте `<header>`. Разделительная линия реализована через псевдоэлемент `after:`, чтобы активная вкладка могла её перекрыть.
*   **Запрещено**: убирать `-mb-px` или `z-10` у активной вкладки -- это сломает эффект бесшовного перехода ("папки").
*   **При добавлении новой страницы**: убедитесь, что она прописана в `breadcrumbMap` (в `app-shell.tsx`) для корректного заголовка вкладки.
*   **При добавлении формы (Smart Component)**:
    1.  Использовать хук `const { markDirty, markClean } = useTabDirty()`.
    2.  Обернуть контейнер формы (или `FormToolbar`) в `onChange={markDirty}`.
    3.  Явно вызывать `markDirty()` при добавлении/удалении строк в деталях (табличных частях).
    4.  Явно вызывать `markClean()` после успешного ответа от API при сохранении/проведении.

## 6. Тестирование

*   **E2E**: Playwright. Основной упор на пользовательские сценарии (Happy Path).
*   **Unit**: Vitest/Jest (для сложной логики утилит и хуков).
