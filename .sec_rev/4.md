# Аудит системы аутентификации — Metapus

## 1. Хеширование паролей — `bcrypt` ✅ с замечанием

**Статус: БЕЗОПАСНО, но cost-фактор минимальный.**

Пароли хешируются через `bcrypt.GenerateFromPassword` с `bcrypt.DefaultCost` (= 10). [1](#0-0) 

MD5/SHA1 для паролей нигде не используется. Refresh-токены хранятся как SHA256-хеш (не пароль — это корректно). [2](#0-1) 

**Замечание:** `bcrypt.DefaultCost = 10` — минимально приемлемо. OWASP рекомендует cost ≥ 12 для современного железа. Фикс:

```go
// internal/domain/auth/service.go, строка 117
passwordHash, err := bcrypt.GenerateFromPassword([]byte(req.Password), 12)
```

## 2. Сессии: срок жизни и инвалидация при логауте ✅

**Статус: БЕЗОПАСНО.**

- Access token: 15 минут (JWT `exp`).
- Refresh token: 7 дней, хранится в БД с `expires_at`. [3](#0-2) 

При логауте вызывается `RevokeAllUserTokens` — все refresh-токены пользователя помечаются `revoked_at = now()`. [4](#0-3) [5](#0-4) 

При каждом refresh старый токен отзывается (ротация), загружаются актуальные права. [6](#0-5)

## 3. Email-верификация — УЯЗВИМОСТЬ ⚠️

**Статус: ПОЛЕ ЕСТЬ, ПРОВЕРКА ПРИ ЛОГИНЕ ОТСУТСТВУЕТ.**

Поле `email_verified` существует в модели: [7](#0-6) 

Но `NewUser()` не устанавливает `EmailVerified = false` явно (Go zero-value = false — это правильно), однако `CanLogin()` **не проверяет** `email_verified`: [8](#0-7) 

**Уязвимость:** Новый пользователь может войти в систему без подтверждения email. Нет механизма отправки верификационного письма.

**Фикс:**
```go
// internal/domain/auth/model.go — добавить в CanLogin()
func (u *User) CanLogin() error {
    if !u.IsActive {
        return apperror.NewForbidden("account is disabled")
    }
    if !u.EmailVerified {
        return apperror.NewForbidden("email not verified")
    }
    if u.IsLocked() {
        return apperror.NewForbidden("account is temporarily locked")
    }
    return nil
}
```

Также необходимо реализовать эндпоинт `/auth/verify-email` с одноразовым токеном и отправку письма при регистрации.

## 4. Токены сброса пароля — ФУНКЦИОНАЛ ОТСУТСТВУЕТ ⚠️

**Статус: НЕТ РЕАЛИЗАЦИИ.**

В `handlers/auth.go` нет эндпоинтов `/auth/forgot-password` и `/auth/reset-password`. [9](#0-8) 

В `auth.Service` нет методов `ForgotPassword` / `ResetPassword`. [10](#0-9) 

**Уязвимость:** Пользователи не могут самостоятельно сбросить пароль. Если функционал будет добавлен без должной реализации — риск небезопасных токенов.

**Требования к реализации:**
- Токен: `crypto/rand`, 32 байта, хранить SHA256-хеш в БД.
- TTL: 1 час (`expires_at = now() + 1h`).
- Одноразовость: после использования — немедленный `revoked_at = now()`.
- Не раскрывать существование email в ответе (`200 OK` всегда).

## 5. Rate Limiting на login-эндпоинте — ЧАСТИЧНО ⚠️

**Статус: ACCOUNT LOCKOUT ЕСТЬ, HTTP RATE LIMITING ПО IP — НЕТ.**

Есть защита на уровне аккаунта: 5 неверных попыток → блокировка на 15 минут: [11](#0-10) [12](#0-11) 

**Уязвимость:** Атакующий может перебирать **разные аккаунты** с одного IP без ограничений. Account lockout не защищает от credential stuffing.

В router.go нет rate limiting middleware на `/auth/login`: [13](#0-12) 

**Фикс** — добавить `golang.org/x/time/rate` или `github.com/ulule/limiter` на публичные auth-маршруты:

```go
// internal/infrastructure/http/v1/router.go
publicAuth.Use(middleware.RateLimit(5, time.Minute)) // 5 req/min per IP
publicAuth.POST("/login", authHandler.Login)
```

## Дополнительные находки

### КРИТИЧНО: Токены в localStorage (XSS-уязвимость) ⚠️

`useAuthStore` использует Zustand `persist` — токены сохраняются в `localStorage`: [20](#0-19) 

**Уязвимость:** Любой XSS-скрипт может прочитать `localStorage.getItem("metapus-auth")` и украсть access + refresh токены.

**Фикс:** Перенести refresh token в `httpOnly; Secure; SameSite=Strict` cookie. Access token можно держать в памяти (не в localStorage).

### СРЕДНЕЕ: JWT-токен в URL для WebSocket ⚠️

Auth middleware принимает токен из query-параметра `?token=`: [21](#0-20) 

**Уязвимость:** Токен попадает в access-логи nginx/сервера, в историю браузера, в `Referer` заголовок.

**Фикс:** Для WebSocket использовать первое сообщение по протоколу для передачи токена, или `Sec-WebSocket-Protocol` header.

### СРЕДНЕЕ: WebSocket CORS — fail-open в production ⚠️

Если `APP_ALLOWED_ORIGINS` не задан в production — все origins разрешены: [22](#0-21) 

**Фикс:** Изменить логику: если `APP_ENV != development` и `APP_ALLOWED_ORIGINS` пуст — запрещать все origins.

### НИЗКОЕ: RLS fail-open для пользователей без профиля

Если у пользователя нет Security Profile — `DataScope{}` пустой, нет ограничений по организациям. Это задокументированное поведение, но потенциально опасно при ошибках конфигурации. [23](#0-22)

```go
func (r *TokenRepo) RevokeAllUserTokens(ctx context.Context, userID id.ID, reason string) error {
	q := r.getTxManager(ctx).GetQuerier(ctx)

	query := `UPDATE refresh_tokens SET revoked_at = now(), revoked_reason = $2 WHERE user_id = $1 AND revoked_at IS NULL`
	_, err := q.Exec(ctx, query, userID, reason)
	if err != nil {
		return fmt.Errorf("revoke all tokens: %w", err)
	}

	return nil
```

```markdown
### Семантика fail-open / fail-closed

| Ситуация | Результат |
|----------|----------|
| Admin | Полный доступ (`DataScope{IsAdmin: true}`) |
| Нет профиля (nil) | **Fail-open** — пустой `DataScope{}`, нет ограничений |
| Ошибка загрузки профиля | **Fail-closed** — пустые `AllowedIDs` = нет доступа |
| Профиль без dimension X | Нет ограничений по dimension X |
| Профиль с пустым `AllowedIDs` | Нет доступа по этому dimension |
```
