# Отчёт по аудиту безопасности Metapus

---

## 🔴 КРИТИЧНО

---
### 1. JWT-токен передаётся через URL query param

**Файл:** `internal/infrastructure/http/v1/middleware/auth.go:33-35`  
**Проблема:** Токен принимается из `?token=` query параметра. [3](#0-2) 

**Риск:** Токен попадает в: access-логи nginx/сервера, browser history, Referer-заголовки при переходе на внешние ресурсы, аналитику. Для WebSocket это допустимо, но нужна изоляция.  
**Фикс:** Для WebSocket использовать одноразовый ticket-токен (короткоживущий, хранится в Redis/БД), а не основной JWT:
```go
// Только для WS: принять ticket, обменять на сессию
if c.FullPath() == "/api/v1/ws" {
    ticket := c.Query("ticket")
    // validate ticket from DB, not JWT
}
```

---

### 2. Внутренние эндпоинты Updater Agent без аутентификации

**Файл:** `internal/infrastructure/http/v1/router.go:602-614`  
**Проблема:** Эндпоинты `/internal/tenants/:id/trigger-update`, `/retry-update`, `/rollback-update` зарегистрированы без какой-либо аутентификации. [4](#0-3) 

**Риск:** Если Docker-сеть неправильно настроена или сервер доступен напрямую (misconfigured reverse proxy), злоумышленник может инициировать откат/обновление любого тенанта, вызвав DoS или даунгрейд до уязвимой версии.  
**Фикс:** Добавить shared secret middleware:
```go
func InternalSecret(secret string) gin.HandlerFunc {
    return func(c *gin.Context) {
        if c.GetHeader("X-Internal-Secret") != secret {
            c.AbortWithStatus(http.StatusForbidden)
            return
        }
        c.Next()
    }
}
```

### 3. Отсутствует rate limiting на аутентификацию и регистрацию

**Файл:** `internal/infrastructure/http/v1/router.go:229-248`, `internal/domain/auth/service.go`  
**Проблема:** Нет IP-based rate limiting на `/auth/login`, `/auth/register`, `/auth/refresh`. Есть account lockout после 5 попыток, но он привязан к аккаунту, а не к IP. [5](#0-4) 

**Риск:** Атака credential stuffing на разные аккаунты с одного IP не блокируется. Атака на `/auth/register` позволяет создавать тысячи аккаунтов. Атака на `/auth/refresh` позволяет перебирать токены.  
**Фикс:** Добавить middleware с `golang.org/x/time/rate` или `github.com/ulule/limiter`:
```go
// 5 попыток в минуту с одного IP
authLimiter := limiter.New(store, limiter.Rate{Period: time.Minute, Limit: 5})
publicAuth.Use(mgin.NewMiddleware(authLimiter))
```

### 4. SMTP без принудительного TLS

**Файл:** `internal/core/automation/adapters.go:229-231`  
**Проблема:** `smtp.SendMail` использует STARTTLS только если сервер его предлагает, но не требует его принудительно. Нет проверки TLS-сертификата. [6](#0-5) 

**Риск:** SMTP-трафик (включая пароли через `PlainAuth`) может передаваться в открытом виде при MITM-атаке или если SMTP-сервер не поддерживает STARTTLS.  
**Фикс:** Использовать явный TLS:
```go
tlsConfig := &tls.Config{ServerName: smtpHost}
conn, err := tls.Dial("tcp", addr, tlsConfig)
// или проверять, что STARTTLS был согласован
```

## 🟠 ВАЖНО

### 5. Отсутствуют HTTP Security Headers

**Файл:** `internal/infrastructure/http/v1/router.go`, `frontend/next.config.mjs`  
**Проблема:** Нет ни одного security header: `X-Frame-Options`, `X-Content-Type-Options`, `Strict-Transport-Security`, `Content-Security-Policy`, `Referrer-Policy`. [7](#0-6) 

**Риск:** Clickjacking, MIME-sniffing атаки, отсутствие HSTS означает возможность downgrade до HTTP.  
**Фикс:** В `next.config.mjs`:
```js
async headers() {
  return [{
    source: '/(.*)',
    headers: [
      { key: 'X-Frame-Options', value: 'DENY' },
      { key: 'X-Content-Type-Options', value: 'nosniff' },
      { key: 'Strict-Transport-Security', value: 'max-age=63072000; includeSubDomains' },
      { key: 'Referrer-Policy', value: 'strict-origin-when-cross-origin' },
    ],
  }]
}
```
На Go-сервере добавить middleware с аналогичными заголовками.

### 6. JWT-токены хранятся в `localStorage` (уязвимость к XSS)

**Файл:** `frontend/stores/useAuthStore.ts:71-79`  
**Проблема:** Zustand persist сохраняет `accessToken` и `refreshToken` в `localStorage`. [8](#0-7) 

**Риск:** Любой XSS-скрипт на странице получает полный доступ к токенам и может действовать от имени пользователя.  
**Фикс:** `accessToken` хранить только в памяти (Zustand без persist), `refreshToken` — в `httpOnly` cookie. Это требует изменения flow обновления токена на серверный endpoint.

---

### 7. `document.write(html)` с серверным HTML — потенциальный XSS

**Файл:** `frontend/components/shared/print-menu-button.tsx:58-60`  
**Проблема:** HTML печатной формы, полученный с сервера, вставляется через `document.write`. [9](#0-8) 

**Риск:** Если шаблон печатной формы содержит пользовательские данные без экранирования (например, имя контрагента с `<script>`), это приведёт к XSS в новом окне браузера.  
**Фикс:** Использовать `srcdoc` в `<iframe>` с `sandbox` атрибутом:
```js
const iframe = document.createElement('iframe')
iframe.setAttribute('sandbox', 'allow-same-origin allow-scripts')
iframe.srcdoc = html
document.body.appendChild(iframe)
```

---

### 8. Fail-open в `SecurityContext` middleware

**Файл:** `internal/infrastructure/http/v1/middleware/security_context.go:73-75`  
**Проблема:** Если пользователю не назначен security profile, он получает **полный доступ** ко всем данным. [10](#0-9) 

**Риск:** Новый пользователь без профиля видит данные всех организаций. Это нарушает принцип least privilege.  
**Фикс:** Изменить поведение на fail-closed (пустой DataScope = нет доступа) или явно требовать назначения профиля при создании пользователя.

### 9. `AUTOMATION_ENCRYPTION_KEY` отсутствует в `.env.example`

**Файл:** `single-tenant-deployment/.env.example`, `internal/infrastructure/storage/postgres/automation_account_repo.go:32-34`  
**Проблема:** Ключ шифрования credentials (Telegram bot tokens, SMTP passwords) не задокументирован в примере конфигурации. [11](#0-10) 

**Риск:** При деплое без этой переменной создание automation accounts завершится ошибкой 500. Если ключ задан неправильной длины (не 32 байта) — то же самое. Нет валидации при старте сервера.  
**Фикс:** Добавить в `.env.example`:
```
# 32 байта для AES-256: openssl rand -hex 16
AUTOMATION_ENCRYPTION_KEY=change-me-32-bytes-exactly-here!!
```
И добавить проверку при старте в `main.go` через `mustEnv`.

### 10. `/health/tenants` без аутентификации раскрывает внутреннюю информацию

**Файл:** `internal/infrastructure/http/v1/router.go:129`  
**Проблема:** Эндпоинт статистики тенантов доступен без аутентификации. [12](#0-11) 

**Риск:** Раскрывает количество тенантов, версии схем, статусы — полезная информация для разведки.  
**Фикс:** Перенести `/health/tenants` в защищённую группу с `RequireRole("admin")`.

### 11. Нет идемпотентности для входящих webhook (ЮKassa/Stripe)

**Файл:** Webhook-обработчики не найдены в коде  
**Проблема:** В коде нет обработчиков входящих webhook от платёжных систем. Если они будут добавлены, общий `Idempotency` middleware не применяется автоматически (он опциональный и требует `X-Idempotency-Key` от клиента). [13](#0-12) 

**Риск:** Повторная доставка webhook (ЮKassa/Stripe гарантируют at-least-once) приведёт к двойному списанию или двойному зачислению.  
**Фикс:** Для webhook-эндпоинтов реализовать дедупликацию по `webhook_id` из тела запроса с хранением в БД:
```go
// Проверить подпись (HMAC-SHA256 от тела)
// Проверить webhook_id в таблице processed_webhooks
// Если уже обработан — вернуть 200 OK без повторной обработки
```

---

### 12. Нет валидации подписи входящих webhook

**Файл:** Webhook-обработчики не найдены  
**Проблема:** Нет кода верификации HMAC-подписи от ЮKassa (`X-Webhook-Signature`) или Stripe (`Stripe-Signature`).  
**Риск:** Злоумышленник может отправить поддельный webhook и инициировать зачисление средств.  
**Фикс:** Обязательно верифицировать подпись до обработки тела запроса.

---

### 13. Нет ограничения на размер тела запроса глобально

**Файл:** `cmd/server/main.go:184-190`  
**Проблема:** `http.Server` не имеет `MaxHeaderBytes`, а Gin не ограничивает тело запроса глобально. Ограничение в 1MB есть только в idempotency middleware. [14](#0-13) 

**Риск:** DoS через отправку огромных тел запросов.  
**Фикс:**
```go
server := &http.Server{
    MaxHeaderBytes: 1 << 20, // 1MB
    Handler: http.MaxBytesHandler(router, 10<<20), // 10MB max body
}
```

---

### 14. SQL-запросы с `fmt.Sprintf` для динамических имён таблиц/колонок

**Файл:** `internal/infrastructure/storage/postgres/ref_finder_repo.go:307-318`  
**Проблема:** `spec.tableName` и `spec.dbColumn` подставляются через `fmt.Sprintf` в SQL. [15](#0-14) 

**Риск:** Сейчас значения берутся из metadata registry (статические), но если registry когда-либо будет принимать данные из API (динамические entity), это станет SQL injection. Аналогичная ситуация в `catalog_repo/base.go:517` (GetTree/GetPath).  
**Фикс:** Добавить whitelist-валидацию имён таблиц и колонок через regexp `^[a-z_]+$` перед подстановкой.

### 17. CORS: `Access-Control-Allow-Credentials: true` при возможном wildcard

**Файл:** `internal/infrastructure/http/v1/middleware/cors.go:28` [18](#0-17) 

**Проблема:** Если `CORS_ALLOWED_ORIGINS=*`, то `isAllowed` вернёт `true` для любого origin, и заголовок `Access-Control-Allow-Credentials: true` будет выставлен вместе с конкретным origin. Браузер это примет.  
**Фикс:** Запретить wildcard при включённых credentials:
```go
if a == "*" {
    // никогда не выставлять credentials с wildcard
    continue
}
```
