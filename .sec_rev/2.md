## Уязвимость 1: `GET /health/tenants` — без аутентификации

**Файл:** `internal/infrastructure/http/v1/router.go`

Эндпоинт помечен в коде как "Admin endpoint for tenant stats", но находится в группе `/health` без какого-либо middleware аутентификации. [1](#0-0) 

**Вектор атаки:** Любой внешний пользователь делает `GET /health/tenants` и получает статистику по всем тенантам (ID, статусы, версии схем).

**Фикс:**
```go
// router.go — вынести /tenants в защищённую группу
health.GET("/live", healthHandler.Live)
health.GET("/ready", healthHandler.Ready)
health.GET("/info", healthHandler.Info)
// Убрать из health-группы:
// health.GET("/tenants", healthHandler.TenantsStats)

// Добавить в registerAdminTenantRoutes или отдельную admin-группу:
admin.GET("/health/tenants", healthHandler.TenantsStats)
// (admin-группа уже имеет middleware.RequireRole("admin"))
```

## Уязвимость 2: `GET /settings` и `PATCH /settings/:section` — нет проверки роли admin

**Файл:** `internal/infrastructure/http/v1/handlers/settings.go`

`SettingsHandler` зарегистрирован в `protected`-группе (требует JWT), но **не** требует роли `admin`. Любой аутентифицированный пользователь может читать и изменять системные настройки. [2](#0-1) [3](#0-2) 

**Вектор атаки:** Обычный пользователь делает `PATCH /api/v1/settings/email` и меняет SMTP-конфигурацию, или `PATCH /api/v1/settings/security` — меняет политики безопасности всего тенанта.

**Фикс:**
```go
// handlers/settings.go
func (h *SettingsHandler) RegisterRoutes(rg *gin.RouterGroup) {
    sg := rg.Group("/settings")
    sg.Use(middleware.RequireRole("admin")) // ← добавить
    {
        sg.GET("", h.Get)
        sg.PATCH("/:section", h.UpdateSection)
    }
}
```

## Уязвимость 3: `/internal/*` — нет аутентификации (сетевое доверие)

**Файл:** `internal/infrastructure/http/v1/router.go`

Эндпоинты Updater Agent (`/internal/tenants/:id/trigger-update`, `/retry-update`, `/rollback-update`, `/migration-status`) не имеют никакой аутентификации. [4](#0-3) 

**Вектор атаки:** Если порт бэкенда доступен не только из Docker-сети (например, при неправильной конфигурации `docker-compose` или облачного файрвола), атакующий может инициировать откат миграции или обновление схемы любого тенанта.

**Фикс:** Добавить проверку shared secret из заголовка:
```go
// middleware/internal_secret.go
func InternalSecret(secret string) gin.HandlerFunc {
    return func(c *gin.Context) {
        if c.GetHeader("X-Internal-Secret") != secret {
            c.AbortWithStatus(http.StatusForbidden)
            return
        }
        c.Next()
    }
}

// router.go
func registerInternalUpdaterRoutes(rg *gin.RouterGroup, cfg RouterConfig, ...) {
    rg.Use(middleware.InternalSecret(cfg.InternalSecret)) // ← добавить
    rg.POST("/tenants/:id/trigger-update", h.InternalTriggerUpdate)
    // ...
}
```

## Уязвимость 4: `NotificationRepo.GetByID` — нет проверки `user_id`

**Файл:** `internal/infrastructure/storage/postgres/notification_repo.go`

Метод `GetByID` выбирает уведомление только по `id`, без проверки принадлежности пользователю. [5](#0-4) 

Для сравнения — `MarkAsRead` правильно добавляет `user_id` в WHERE: [6](#0-5) 

**Вектор атаки:** Если `GetByID` будет использован в новом эндпоинте (например, `GET /system/notifications/:id`), атакующий сможет передать чужой UUID и получить чужое уведомление.

**Фикс:**
```go
// notification_repo.go

        ToSql()
```

## Уязвимость 5: `SecurityContext` middleware — опциональный, без него нет DataScope

**Файл:** `internal/infrastructure/http/v1/router.go`

Middleware `SecurityContext` (который строит DataScope и FieldPolicies для RLS/FLS) применяется **только если** `cfg.ProfileProvider != nil`. [7](#0-6) 

**Вектор атаки:** Если `ProfileProvider` не сконфигурирован (например, в тестовом или self-hosted деплое), DataScope не применяется — все пользователи видят все данные тенанта без ограничений по профилям безопасности.

**Фикс:** Сделать `ProfileProvider` обязательным или добавить явный fallback:
```go
// router.go
if cfg.ProfileProvider == nil {
    panic("v1.NewRouter: cfg.ProfileProvider must not be nil — security profiles are required")
}
protected.Use(middleware.SecurityContext(cfg.ProfileProvider))
```

## Уязвимость 6: Отсутствие RLS-политик в PostgreSQL

**Директория:** `db/migrations/`

Ни в одной миграции не включён `ROW LEVEL SECURITY` и не созданы политики. Таблицы `sys_notifications`, `user_preferences`, `users`, `refresh_tokens` и все бизнес-таблицы не защищены на уровне БД. [8](#0-7) [9](#0-8) 

**Вектор атаки:** При SQL-инъекции, утечке credentials или прямом доступе к БД (например, через pgAdmin) — нет второго уровня защиты. Любой DB-пользователь читает все строки всех таблиц.

**Фикс (пример для `sys_notifications`):**
```sql
-- В миграции 00028 или отдельной миграции:
ALTER TABLE sys_notifications ENABLE ROW LEVEL SECURITY;

CREATE POLICY notifications_owner ON sys_notifications
    FOR ALL
    USING (user_id = current_setting('app.current_user_id')::uuid);
```

И в Go-коде перед каждым запросом устанавливать `SET LOCAL app.current_user_id = $1`.

```go
// requireUserID extracts and parses user ID from context, sending error if absent.
func (h *UserPrefsHandler) requireUserID(c *gin.Context) (id.ID, error) {
	raw := h.GetUserID(c)
	if raw == "" {
		err := apperror.NewUnauthorized("not authenticated")
		h.Error(c, err)
		return id.ID{}, err
	}

	userID, err := id.Parse(raw)
	if err != nil {
		appErr := apperror.NewValidation("invalid user id")
		h.Error(c, appErr)
		return id.ID{}, appErr
	}

	return userID, nil
}
```
