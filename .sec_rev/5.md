### 🚨 [1] Unauthenticated Internal Updater Routes — Arbitrary Tenant Migration Trigger

* **CWE**: CWE-306 (Missing Authentication for Critical Function)
* **Severity**: CRITICAL
* **Локация**: `internal/infrastructure/http/v1/router.go:604-614`

**Описание проблемы:**

Функция `registerInternalUpdaterRoutes` регистрирует четыре мутирующих эндпоинта на группе `/internal` **без какой-либо аутентификации**. Комментарий в коде прямо подтверждает это: *"No auth required — secured by Docker network isolation (internal network trust)"*. Эти эндпоинты позволяют запустить, откатить или повторить миграцию базы данных для **любого тенанта** по его UUID. [1](#0-0) 

**Вектор атаки (Exploit Scenario):**

1. Атакующий получает UUID тенанта через уязвимость #2 (`GET /health/tenants`) или через OSINT.
2. Атакующий находит SSRF-уязвимость в любом другом сервисе в той же Docker-сети (или Docker-сеть оказывается доступна из-за misconfiguration).
3. Атакующий отправляет: `POST http://backend:8080/internal/tenants/{tenantId}/rollback-update`
4. Сервер выполняет откат схемы БД тенанта без какой-либо проверки прав — потенциальная потеря данных и полный DoS для тенанта.
5. Альтернативно: `POST /internal/tenants/{tenantId}/trigger-update` запускает миграцию в произвольный момент, что может сломать работающий тенант.

**Remediation:**

Добавить shared secret или mTLS для internal-эндпоинтов. Минимальный вариант — проверка статического токена из env:

```go
// registerInternalUpdaterRoutes
internal.Use(func(c *gin.Context) {
    secret := os.Getenv("INTERNAL_API_SECRET")
    if secret == "" || c.GetHeader("X-Internal-Secret") != secret {
        c.AbortWithStatus(http.StatusUnauthorized)
        return
    }
    c.Next()
})
```

### 🚨 [2] Unauthenticated Tenant Enumeration via `/health/tenants`

* **CWE**: CWE-200 (Exposure of Sensitive Information to an Unauthorized Actor)
* **Severity**: HIGH
* **Локация**: `internal/infrastructure/http/v1/router.go:129`, `internal/infrastructure/http/v1/handlers/health.go:148-168`

**Описание проблемы:**

Эндпоинт `GET /health/tenants` зарегистрирован в группе `/health` **без middleware аутентификации или авторизации**. Он возвращает полный список всех тенантов с их `tenant_id` (UUID), `db_name`, статистикой соединений и временем последнего использования. Эта информация является критической для атак на тенант-специфичные эндпоинты. [2](#0-1) [3](#0-2) 

**Вектор атаки:**

1. Атакующий отправляет `GET /health/tenants` — без токена, без заголовков.
2. Получает JSON со списком всех `tenant_id` и `db_name` всех клиентов платформы.
3. Использует полученные `tenant_id` для атак на `/api/v1/*` эндпоинты (брутфорс паролей, эксплуатация уязвимости #1, и т.д.).
4. `db_name` раскрывает схему именования баз данных, что упрощает атаки при компрометации PostgreSQL.

**Remediation:**

```go
// Перенести /health/tenants в защищённую группу или добавить IP allowlist:
health.GET("/tenants", middleware.RequireInternalIP(), healthHandler.TenantsStats)
// Либо полностью убрать из публичного роутера и вынести в отдельный admin-порт.
```

### 🚨 [3] JWT Token Exposure via URL Query Parameter

* **CWE**: CWE-598 (Sensitive Information in Query String)
* **Severity**: HIGH
* **Локация**: `internal/infrastructure/http/v1/middleware/auth.go:33-35`

**Описание проблемы:**

Auth middleware принимает JWT-токен не только из заголовка `Authorization`, но и из URL query-параметра `?token=`. Это сделано для поддержки WebSocket-соединений. Однако URL с токеном попадает в: access-логи nginx/Gin, логи reverse proxy, `Referer`-заголовки при редиректах, историю браузера, аналитические системы. [4](#0-3) 

**Вектор атаки:**

1. Пользователь открывает WebSocket: `wss://app.example.com/api/v1/ws?token=eyJhbGci...`
2. Nginx логирует полный URL: `GET /api/v1/ws?token=eyJhbGci... 101`
3. Атакующий с доступом к логам (DevOps, SIEM, compromised log aggregator) извлекает действующий JWT.
4. Токен действителен 15 минут (`AccessTokenTTL: 15 * time.Minute`). За это время атакующий получает полный доступ к аккаунту жертвы.

**Remediation:**

Для WebSocket использовать одноразовый ticket-механизм:

```go
// 1. Клиент запрашивает короткоживущий (30s) ws-ticket через защищённый REST-эндпоинт
// POST /api/v1/ws/ticket → { "ticket": "random-32-bytes", "expiresAt": "..." }
// 2. Ticket хранится в Redis/памяти, привязан к userID
// 3. WebSocket handshake: wss://app/ws?ticket=random-32-bytes
// 4. Сервер обменивает ticket на userID, ticket немедленно инвалидируется
```

### 🚨 [4] Potential SQL Injection via Unvalidated `spec.Field` in `listAround`

* **CWE**: CWE-89 (SQL Injection)
* **Severity**: HIGH (требует верификации)
* **Локация**: `internal/infrastructure/storage/postgres/document_repo/base.go:594`

**Описание проблемы:**

В методе `listAround` значение `spec.Field` интерполируется напрямую в SQL-строку через `fmt.Sprintf` без использования параметризованных запросов. `spec.Field` получается из `keyset.ParseOrderBy(f.OrderBy, ...)`, где `f.OrderBy` — это пользовательский query-параметр `?orderBy=`. Если `keyset.ParseOrderBy` не валидирует значение строго против whitelist `r.orderCols`, это — SQL Injection. [5](#0-4) 

**Вектор атаки:**

1. Атакующий отправляет: `GET /api/v1/document/goods-receipt?cursor=around&targetId=<valid-id>&orderBy=id;SELECT+pg_sleep(5)--`
2. Если `ParseOrderBy` не валидирует, `spec.Field` = `id;SELECT pg_sleep(5)--`
3. Итоговый SQL: `SELECT id;SELECT pg_sleep(5)-- FROM doc_goods_receipts WHERE id = $1`
4. Time-based blind SQLi позволяет извлечь данные из БД тенанта.

**Статус**: Пакет `internal/infrastructure/storage/postgres/keyset` не был прочитан в рамках аудита. **Необходима немедленная верификация** реализации `ParseOrderBy`.

**Remediation:**

Независимо от реализации `ParseOrderBy`, исправить интерполяцию:

```go
// БЫЛО (небезопасно):
lookupSQL := fmt.Sprintf("SELECT %s FROM %s WHERE id = $1", spec.Field, r.tableName)

// СТАЛО (безопасно — column identifier не может быть параметром $N,
// но должен быть явно проверен против whitelist):
if _, ok := r.orderCols[spec.Field]; !ok {
    return result, apperror.NewValidation("invalid sort field")
}
lookupSQL := fmt.Sprintf("SELECT %s FROM %s WHERE id = $1",
    pgx.Identifier{spec.Field}.Sanitize(), // экранирование идентификатора
    pgx.Identifier{r.tableName}.Sanitize())
```

### 🚨 [5] Batch Filter DoS — Authenticated Resource Exhaustion

* **CWE**: CWE-400 (Uncontrolled Resource Consumption)
* **Severity**: MEDIUM
* **Локация**: `internal/infrastructure/http/v1/handlers/document.go:626`, `document.go:687`

**Описание проблемы:**

`BatchActionByFilter` позволяет обработать до `DefaultBatchFilterLimit = 100_000` документов за один запрос. Конкурентность ограничена `maxConnsPerTenant = 10`. Авторизованный пользователь (даже с минимальными правами на `post`) может запустить несколько параллельных запросов, каждый из которых займёт все 10 соединений тенанта на длительное время, полностью блокируя работу других пользователей этого тенанта. [6](#0-5) [7](#0-6) 

**Remediation:**

```go
// 1. Ограничить BatchActionByFilter отдельным, более низким лимитом (например, 1000)
const DefaultBatchFilterLimit = 1000 // вместо 100_000

// 2. Добавить rate limiting на batch-эндпоинты (1 активный batch на пользователя)
// 3. Добавить HTTP-таймаут для batch-операций через context.WithTimeout
```
