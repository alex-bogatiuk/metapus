# Стандарт документации Metapus

## Цель

Создать единый регламент написания технической документации, оптимизированный для трёх потребителей: **разработчик**, **ИИ-агент**, **администратор ERP**. Документ не ссылается на текущее состояние — описывает целевое.

---

## 1. Таксономия документов (Divio + ERP)

Каждый документ принадлежит одному из четырёх типов. Тип определяет структуру, стиль и аудиторию.

| Тип | Вопрос читателя | Стиль | Пример |
|-----|-----------------|-------|--------|
| **Concept** (Объяснение) | «Что это и зачем?» | Повествовательный, с диаграммами | Архитектура, Multi-tenancy, Posting Engine |
| **How-To** (Рецепт) | «Как сделать X?» | Пронумерованные шаги, копипастабельный код | Добавить справочник, Добавить отчёт, Добавить настройку |
| **Reference** (Справочник) | «Какие параметры / поля / API?» | Таблицы, сигнатуры, без прозы | Naming conventions, API endpoints, Permissions |
| **Rule** (Регламент) | «Что обязательно / запрещено?» | Императивный, таблицы ✅/❌ | Development rules, UX-стандарт, Migration rules |

> **Правило:** Один документ = один тип. Если документ содержит и «зачем» и «как» — разделить на два, связав ссылками.

### Маппинг на ERP-аналоги

| Metapus | 1С ИТС | SAP Help | Odoo Docs |
|---------|--------|----------|-----------|
| Concept | Руководство разработчика | Concept Guide | Developer Tutorials |
| How-To | Примеры | Implementation Guide | How-to Guides |
| Reference | Синтаксис-помощник | API Reference | Technical Reference |
| Rule | Стандарты и методологии | Fiori Design Guidelines | Contributing Guide |

## 2. Анатомия документа (шаблон)

Каждый документ состоит из обязательных блоков. Порядок фиксирован.

```markdown
# NN. Название (EN subtitle)                    ← H1, единственный

> **TL;DR:** Одно предложение — что, зачем, ключевое свойство.

> **Тип:** Concept | How-To | Reference | Rule
> **Аудитория:** Developer | AI-Agent | Admin | All
> **Связанные:** [NN-xxx](NN-xxx.md), [MM-yyy](MM-yyy.md)

---

## 1. [Первая секция]
...содержание...

## N. Файловая карта                              ← обязательно
```path/to/file — краткое описание роли файла
```

## N+1. Чеклист                                   ← для How-To
- [ ] Шаг 1
- [ ] Шаг 2

## N+2. Связанные документы                       ← всегда последний
- [NN-xxx](NN-xxx.md) — одно предложение зачем
```

### Обязательные блоки по типу

| Блок | Concept | How-To | Reference | Rule |
|------|:-------:|:------:|:---------:|:----:|
| TL;DR | ✅ | ✅ | ✅ | ✅ |
| Метаданные (тип, аудитория) | ✅ | ✅ | ✅ | ✅ |
| Диаграмма / схема | ✅ | — | — | — |
| Пронумерованные шаги | — | ✅ | — | — |
| Таблицы API / полей | — | — | ✅ | — |
| Таблицы ✅/❌ | — | — | — | ✅ |
| Файловая карта | ✅ | ✅ | — | — |
| Чеклист | — | ✅ | — | — |
| Связанные документы | ✅ | ✅ | ✅ | ✅ |

## 3. Стиль текста (UX-редактор для документации)

### 3.1. Принцип сканирования

Документацию **сканируют**, а не читают. Каждый абзац должен быть понятен при беглом просмотре.

| Правило | ✅ | ❌ |
|---------|---|---|
| Заголовок = существительное или именная группа | «Курсорная пагинация» | «Как работает пагинация» |
| Первое предложение секции = факт | «Курсор — base64 JSON с последним значением сортировки.» | «В этом разделе мы рассмотрим...» |
| Один абзац = одна мысль, ≤3 предложения | — | Стена текста на 10 строк |
| Таблица вместо перечисления, если ≥3 колонки | Таблица `Поле / Тип / Описание` | Буллет-лист с описаниями |
| Код вместо описания, если можно показать | ```go\ntype X struct{}``` | «Структура X содержит поля...» |

### 3.2. Голос и тон

| Аспект | Правило |
|--------|---------|
| Лицо | Безличный или «мы» (коллективное) |
| Время | Настоящее: «Система создаёт», не «Система будет создавать» |
| Модальность | Факт, не предположение: «Движения не обновляются», не «Движения не должны обновляться» |
| Вежливость | Нулевая. Без «пожалуйста», «рекомендуется», «желательно» |
| Жаргон | Допустим только определённый в глоссарии. Первое упоминание — с расшифровкой |

### 3.3. Запрещённые конструкции

| ❌ Запрещено | ✅ Замена |
|-------------|----------|
| «В данном разделе рассматривается...» | Удалить. Начать с факта. |
| «Как было сказано выше...» | Ссылка: «см. [§2](#2-...)» |
| «Важно отметить, что...» | `> [!IMPORTANT]` alert |
| «Например, ...» (без кода) | Код-блок с примером |
| «и т.д.», «и прочее» | Явное перечисление или «+ N других» |
| Пассивный залог без субъекта | Указать субъект: «Compiler генерирует SQL» |

## 4. Оптимизация для ИИ-агента

ИИ-агент — полноценный потребитель документации. Он ищет по ключевым словам, парсит структуру и принимает решения на основе найденного.

### 4.1. ROUTER.md — точка входа

Единственный файл, который агент читает первым. Требования:

| Свойство | Зачем |
|----------|-------|
| Таблица `# / Файл / Описание / Ключевые слова` | Агент находит релевантный документ по ключевым словам |
| Секция «Я хочу...» | Маппинг задачи → документ (intent-based routing) |
| Иерархия чтения | Порядок для onboarding нового агента |

### 4.2. Машиночитаемые паттерны

| Паттерн | Пример | Зачем |
|---------|--------|-------|
| **Файловая карта** в каждом документе | `internal/domain/.../model.go — Domain struct` | Агент знает, какой файл открыть |
| **Чеклист** для How-To | `- [ ] Миграция` | Агент использует как task list |
| **Дерево решений** (ASCII) | `├── Влияет на систему? → sys_settings` | Агент парсит ветвления |
| **Таблицы** вместо прозы | `| Поле | Тип | Описание |` | Структурированный парсинг |
| **Явные пути к файлам** | `internal/domain/settings/model.go` | Прямой переход без поиска |
| **Связанные документы** | Ссылки в конце каждого документа | Граф знаний для навигации |

### 4.3. Ключевые слова в ROUTER.md

Каждый документ имеет набор ключевых слов — **тегов**, по которым агент находит его при поиске. Теги должны включать:

- Технические термины (`pgx`, `squirrel`, `JSONB`)
- Доменные понятия (`проведение`, `остатки`, `справочник`)
- Английские эквиваленты (`posting`, `stock`, `catalog`)
- Синонимы и смежные термины (`register`, `ledger`, `movement`)

## 5. Код в документации

### 5.1. Правила оформления

| Правило | Пример |
|---------|--------|
| Первая строка блока = путь к файлу (комментарий) | `// internal/domain/settings/model.go` |
| Язык указан в fence | ` ```go `, ` ```typescript `, ` ```sql ` |
| Только уникальный код, без boilerplate | Показать `Validate()`, не весь `model.go` |
| Пропуски обозначены `// ...existing...` | Контекст без шума |
| Compile-ready: агент может скопировать и вставить | Никаких `...`, `/* TODO */` без пояснения |

### 5.2. Примеры vs Шаблоны

| Тип | Когда | Именование переменных |
|-----|-------|----------------------|
| **Пример** (конкретный) | Concept, Reference | `GoodsReceipt`, `StockBalance` — реальные сущности |
| **Шаблон** (generic) | How-To | `YourEntity`, `your_entities` — placeholder'ы |

## 6. Визуальные элементы

### 6.1. Диаграммы (Mermaid / ASCII)

| Тип диаграммы | Когда |
|---------------|-------|
| ASCII box-diagram | Архитектурные слои, pipeline'ы, потоки данных |
| Mermaid flowchart | Деревья решений, state machines |
| Mermaid sequence | Request lifecycle, posting flow |
| Таблица | Всё остальное (не рисуй, если можно табличить) |

### 6.2. Alerts (GitHub-style)

| Alert | Когда |
|-------|-------|
| `> [!NOTE]` | Дополнительный контекст, не критичный |
| `> [!TIP]` | Оптимизация, best practice |
| `> [!IMPORTANT]` | Архитектурное решение, которое нельзя нарушать |
| `> [!WARNING]` | Breaking change, совместимость |
| `> [!CAUTION]` | Потеря данных, безопасность |

**Правило:** Не более 2 alerts на секцию. Не ставить подряд.

## 7. Навигация и перекрёстные ссылки

### 7.1. Правила ссылок

| Правило | ✅ | ❌ |
|---------|---|---|
| Ссылка = `[NN-name.md](NN-name.md)` | `[14-howto-new-entity.md](14-howto-new-entity.md)` | `[тут](14-howto-new-entity.md)` |
| При первом упоминании — с пояснением | «см. [14-howto-new-entity.md](14-howto-new-entity.md) (пошаговое руководство)» | Голая ссылка |
| Ссылка на секцию — через якорь | `[§3.2](#32-поля)` | «Смотри выше» |
| Дублирование в «Связанные документы» | Все упомянутые документы собраны в конце | Только inline |

### 7.2. Секция «Связанные документы»

Последняя секция каждого документа. Формат:

```markdown
## Связанные документы

- [09-crud-pipeline.md](09-crud-pipeline.md) — generic CRUD: handler → service → repo
- [14-howto-new-entity.md](14-howto-new-entity.md) — пошаговое добавление сущности
```

Каждая ссылка содержит **одно предложение** — зачем читать этот документ.

| Правило | Пример |
|---------|--------|
| Префикс `NN-` (двузначный номер) | `09-crud-pipeline.md` |

## 9. Глоссарий (обязательные термины)

Документация использует фиксированный набор терминов. Синонимы запрещены.

| Термин | Значение | ❌ Не использовать |
|--------|----------|-------------------|
| Справочник | `entity.Catalog` | Мастер-данные, словарь, классификатор |
| Документ | `entity.Document` | Транзакция, операция |
| Движение | Запись в регистре | Проводка, запись |
| Проведение | `Engine.Post()` | Фиксация, подтверждение |
| Секция (настроек) | JSONB-колонка в `sys_settings` | Раздел, модуль, категория |
| Реестр (FactoryRegistry) | Контейнер регистраций | Регистр (это другое), хранилище |
| Регистр (накопления) | `reg_*` таблица | Реестр, журнал |
| Хук | Функция в `HookRegistry` | Callback, подписка, обработчик |

## 10. Жизненный цикл документа

### 10.1. Когда создавать

| Триггер | Действие |
|---------|----------|
| Новая архитектурная подсистема | Concept документ |
| Новый тип сущности (паттерн) | How-To документ |
| Новый набор API / конфигураций | Reference документ |
| Новое архитектурное ограничение | Добавить в существующий Rule |

### 10.2. Когда обновлять

| Триггер | Действие |
|---------|----------|
| Изменился API / структура | Обновить Reference |
| Изменился процесс добавления | Обновить How-To |
| Изменилось архитектурное решение | Обновить Concept + Rule |
| Добавлен файл в подсистему | Обновить файловую карту |

### 10.3. Верификация

При каждом PR, затрагивающем документацию:

- [ ] Заголовок — существительное, именительный падеж
- [ ] TL;DR — одно предложение, факт
- [ ] Тип документа указан
- [ ] Файловая карта актуальна
- [ ] Все ссылки рабочие
- [ ] Нет запрещённых конструкций (§3.3)
- [ ] Глоссарий соблюдён (§9)
- [ ] ROUTER.md обновлён

## Open Questions

> [!IMPORTANT]
> 1. **Нужен ли отдельный файл `GLOSSARY.md`** или достаточно секции в этом стандарте? Глоссарий будет расти — отдельный файл удобнее для поиска, но создаёт ещё один документ для поддержки.
> 2. **Стоит ли добавить тип `Tutorial`** (пошаговое обучение с контекстом) отдельно от How-To (безконтекстный рецепт)? Для ERP-системы с обучением новых разработчиков это может быть полезно.
> 3. **Нужно ли версионировать документацию** (changelog в каждом документе) или достаточно git history?
