# Задание 1. Описание тематики дипломного проекта

**Тема:** «Проектирование и разработка многопользовательской ERP-системы с физической изоляцией данных (Database-per-Tenant) и генерацией интерфейса на основе метаданных»

---

## 1. Актуальность темы

В рамках импортозамещения программного обеспечения разработка отечественных облачных систем планирования ресурсов предприятия (ERP) является востребованным направлением. Большинство существующих SaaS-решений используют логическое разделение данных в рамках единой базы данных (Shared Database), что несет в себе риски утечки информации из-за возможных ошибок в прикладном коде или SQL-запросах.

Разрабатываемая система **Metapus** устраняет этот класс уязвимостей за счет использования архитектуры физической изоляции баз данных (**Database-per-Tenant**). Это гарантирует независимое хранение информации каждого клиента и упрощает соответствие требованиям законодательства (в частности, ФЗ-152 «О персональных данных»).

Дополнительно в проекте решается проблема снижения трудозатрат на разработку интерфейсов при изменении бизнес-требований. Подход **Code-is-Metadata** (динамическая генерация веб-интерфейса на основе структур данных бэкенда) исключает необходимость ручного проектирования форм и снижает вероятность рассогласования программных контрактов.

Использование компилируемого языка **Go 1.25** на бэкенде и СУБД **PostgreSQL 17** позволяет достичь необходимых показателей производительности и масштабируемости при обработке транзакций.

---

## 2. Целевая аудитория

Разрабатываемая ERP-система ориентирована на следующие сегменты:
- **Малый и средний бизнес (SME):** торговые и сервисные компании, нуждающиеся в масштабируемой системе учета с низкими затратами на администрирование.
- **Организации сферы электронной коммерции:** компании с требованиями к надежности проведения транзакций и разделению балансов контрагентов.
- **Системные интеграторы:** разработчики отраслевых ИТ-решений, использующие ядро ERP в качестве платформы для создания собственных продуктов через механизм плагинов.

---

## 3. Предполагаемые функциональные возможности

- **Физическая мультитенантность:** автоматическое развертывание и применение миграций схемы изолированных баз данных для каждого нового арендатора (тенанта) в реальном времени.
- **Управление доступом (RBAC/FLS/RLS):** разграничение прав пользователей на уровне ролей, строк (записей) и отдельных полей таблиц с использованием интерпретатора CEL-политик (Common Expression Language).
- **Двойная запись (Immutable Ledger):** регистрация складских и финансовых операций в регистрах без возможности прямого изменения или удаления существующих движений. Корректировка данных выполняется только методом сторнирования.
- **Динамический интерфейс:** автоматический рендеринг таблиц, экранных форм ввода и панелей фильтрации на стороне клиента на основе JSON-метаданных, передаваемых сервером.

---

## 4. Используемые технологии

- **Backend:** Go 1.25, Gin Gonic (HTTP-роутер), pgx v5 и squirrel (взаимодействие с PostgreSQL), Goose (управление миграциями), Google CEL-go (контроль политик доступа).
- **Frontend:** Next.js 16 (App Router), React 19, TypeScript, Tailwind CSS, компонентная база shadcn/ui, Zustand (управление состоянием), React Hook Form и Zod (валидация данных).
- **Инфраструктура:** Docker и Docker Compose (контейнеризация сервисов), PostgreSQL 17 (система управления базами данных, UUIDv7).

---

## 5. Ожидаемый результат

Рабочий прототип ядра многопользовательской ERP-системы, функционирующий по модели SaaS:
1. **Серверный программный интерфейс (Backend API):** веб-служба на Go с разделением ответственности по Clean Architecture, транзакционным движком проведения документов и изоляцией данных.
2. **Клиентское приложение (Frontend):** веб-приложение на Next.js с поддержкой вкладок, защитой от потери несохраненных данных (dirty-state) и генерацией страниц на основе метаданных.
3. **Система безопасности:** настроенные правила изоляции БД и гибкая ролевая модель разграничения прав.
4. **Инфраструктурные скрипты:** конфигурационные файлы для развертывания и тестирования среды в Docker.
