# Итоги разработки: Архитектура ABAC (Attribute-Based Access Control)

В рамках данной сессии был спроектирован и реализован фундаментальный слой авторизации и разделения прав доступа (RLS и FLS) для ERP-системы Metapus.

## Основные этапы и архитектурные решения

### 1. Реализация универсального Row-Level Security (RLS)
Изначально был предложен вариант с жестко заданными полями (организация и контрагент), однако по требованию к масштабируемости архитектура была переработана на **универсальную систему измерений (Dimensions)**.

*   **Структура `DataScope` (`internal/core/security/data_scope.go`)**:
    *   Ядро очищено от бизнес-логики. Вместо хардкода колонок используется универсальная мапа `Dimensions map[string][]string` (где ключ — логическое имя измерения, например `organization`, `cost_article`, `project`, а значение — массив разрешенных ID).
    *   Реализованы методы генерации SQL-условий (`ApplyConditions`) и point-check проверок для загруженных из БД записей (`CanAccessRecord`).
*   **Интеграция с базовыми репозиториями (`BaseCatalogRepo`, `BaseDocumentRepo`)**:
    *   Добавлен метод `RegisterRLSDimension("dimension_name", "db_column")` для декларативного маппинга логических измерений безопасности на реальные колонки БД.
    *   В универсальный генератор SQL-запросов (`buildWhereConditions`) внедрена автоматическая подстановка `WHERE column IN (...)` на основе контекста пользователя. Расширение прав на новые справочники теперь требует добавления ровно 1 строки кода в конструктор репозитория.
*   **Очистка ядра**:
    *   Доступ к организациям (`OrgIDs`) оставлен на базовом уровне (JWT, `UserContext`).
    *   Доступ к контрагентам и другим будущим разрезам абстрагирован. Предполагается, что они будут загружаться из отдельной подсистемы Security Profiles (БД/Кэш) на этапе Middleware.

### 2. Реализация Field-Level Security (FLS)
Создан механизм для ограничения доступа к отдельным полям документов (например, сокрытие закупочных цен) без дублирования DTO.

*   **`FieldPolicy` (`internal/core/security/field_policy.go`)**:
    *   Реализован мини-DSL для белых и черных списков полей (поддержка `*`, `-exclusions`, и явных списков `["quantity", "price"]`).
    *   Поддержка политик для табличных частей (Table Parts).
*   **`FieldMasker` (`internal/core/security/field_masker.go`)**:
    *   Реализован подход "Approach B" (`ValidateFieldChanges`): при редактировании документа система сравнивает старое и новое состояние и блокирует операцию **только в том случае**, если пользователь попытался изменить поле, к которому у него нет доступа. Это позволяет клиенту свободно отправлять полные DTO.
    *   Реализована функция `MaskForRead` для обнуления запрещенных к чтению полей перед отправкой клиенту.
    *   **Оптимизация**: Для обхода структур используется reflection с **кэшированием метаданных полей** (`globalFieldCache`), что исключает penalty на производительность (reflection вызывается только один раз на тип сущности).

### 3. Тестирование и верификация
*   Написаны исчерпывающие unit-тесты для `DataScope`, `FieldPolicy` и `FieldMasker` (`..._test.go`), покрывающие edge-кейсы (пустые политики, отсутствие пересечений, wildcard-исключения, пересечение нескольких RLS-измерений).
*   Код успешно проходит сборку (`go build ./...`) и все тесты (`go test`).

## Следующие шаги (Next Steps)
Текущая работа заложила инфраструктуру. Для полноценного функционирования потребуется:
1.  Создание БД-структуры и сервиса для управления **Security Profiles** (для хранения прав по контрагентам, статьям ДДС и т.д.).
2.  Написание **HTTP Middleware**, который будет собирать JWT-токен и Security Profile пользователя в единый `DataScope` и класть в context.
3.  Внедрение `DataScope` в сервисный слой (проброс в `ListFilter` при выборках, проверка прав при точечных мутациях).
4.  Интеграция `FieldMasker` в HTTP-хендлеры (маскировка при выдаче, валидация при сохранении).
5.  Внедрение Google CEL (Phase 3) для сложных составных бизнес-правил (например, "доступ только к неоплаченным документам до 100 тыс. руб").
