# Metapus — Design Patterns Rule

## Цель

В проекте Metapus используем **переиспользуемые архитектурные паттерны**.
При добавлении нового кода **предпочитай расширение существующих абстракций**, а не написание новой логики с нуля.

---

## Базовые паттерны проекта

### 1) Factory Pattern для API-слоя
- Используем фабрики API-клиентов вместо ручного дублирования методов:
  - `createCatalogApi<TRes, TCreate, TUpdate>()`
  - `createDocumentApi<TRes, TCreate, TUpdate>()`
- Новые сущности подключаются через конфигурацию фабрики, а не через копипасту CRUD-методов.

### 2) Generic Page Components (Configuration over Boilerplate)
- Списки справочников строятся через generic-компоненты:
  - `CatalogListPage`
  - `ListContent`
- Поведение страницы задается колонками, маршрутом и fetcher-логикой, а не уникальной версткой для каждой сущности.

### 3) Custom Hooks как слой orchestration
- Для форм и списков используем специализированные хуки:
  - `useCatalogForm`
  - `useDocumentListPage`
  - `useFormDraft`
  - `useCollapsible`
- Хуки инкапсулируют загрузку, сохранение, draft, dirty-state, фильтрацию и UI-состояние.

### 4) Shared Components для документных форм
- Общие части документных форм выносятся в shared-слой:
  - `lib/document-form.ts` (общие типы и вычисления)
  - `DocumentTotalsFooter`
- Дублирующую логику строк/итогов не копируем по страницам `new/[id]`.

### 5) Metadata-driven Filtering
- Фильтры списков строятся от metadata (`useEntityFiltersMeta`) и `buildFilterItems()`.
- Не хардкодим схему фильтров, если можно получить её из backend metadata.

### 6) Composition over Copy-Paste
- Новую функциональность добавляем через композицию существующих хуков/компонентов.
- Допускается новый код только если:
  1. Нет подходящей существующей абстракции,
  2. Расширение старой абстракции ухудшает читаемость,
  3. Нужна явно отдельная бизнес-логика.

### 7) Strict Typing (TypeScript-first)
- Не используем `any` в новых фичах.
- DTO и маппинги (`mapFromResponse`, `mapToCreate`, `mapToUpdate`) должны быть явно типизированы.
- Для денег/сумм учитываем валютный scale (`decimalPlaces`) — без хардкода `100`.

---

## Практическое правило для новых задач

Перед добавлением нового экрана/формы/списка:
1. Найди существующий аналог.
2. Проверь, можно ли решить задачу через уже существующий hook/component/factory.
3. Если создаешь новый паттерн — сделай его generic и reusable, не под одну страницу.
4. Проверь `npx tsc --noEmit` и убедись, что нет регрессий.

---

## Anti-patterns (чего избегать)

- Копипаста CRUD-методов и page-логики между сущностями.
- Дублирование вычислений сумм/НДС в нескольких формах.
- Локальные ad-hoc решения вместо расширения shared hooks/components.
- Хардкод денежных scale/форматирования.
- Слабая типизация API-данных и форм.
