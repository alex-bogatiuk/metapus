## Концепция: Отчёт как первоклассный объект метаданных

Архитектура Metapus уже имеет иерархию объектов: `Catalog → Document → Register → Report`. [1](#6-0) 

Но сейчас `ReportRegistration` — это просто `RouteRegistration` с HTTP-маршрутами. Идеальное решение делает отчёт **полноценным объектом системы** с метаданными, типизированным контрактом и интеграцией с security. [2](#6-1)

│          Компоновки Данных) — декларативная схема:

│          декларативная схема, платформа берёт на себя
│          SQL, группировки, итоги, экспорт.

## Идеальный backend

### 1. Типизированный generic интерфейс

```go
// internal/platform/report_contract.go

// ReportRegistration — типизированный контракт отчёта.
// F — тип фильтра, R — тип результата.
// Платформа автоматически:
//   - регистрирует GET /{prefix} endpoint
//   - применяет RequirePermission(Permission())
//   - применяет DataScope (RLS) из контекста
//   - регистрирует GET /{prefix}/export endpoint
//   - отдаёт Meta() через GET /metadata/reports/{prefix}
type ReportRegistration[F any, R any] interface {
    RoutePrefix() string
    Permission() string
    Meta() ReportMeta
    Execute(ctx context.Context, filter F) (R, error)
}

// ReportMeta — декларативное описание отчёта.
// CODE IS METADATA — единственный источник истины.
type ReportMeta struct {
    Name          string
    Description   string
    Filters       []ReportFilterDef   // metadata-driven фильтры
    Columns       []ReportColumnDef   // колонки с типами и форматированием
    GroupBy       []ReportGroupByDef  // доступные группировки
    Totals        []ReportTotalDef    // какие колонки суммировать
    ExportFormats []string            // ["csv", "xlsx"]
    // DataScope dimensions для RLS
    ScopeDimensions []string          // ["warehouse", "organization"]
}
```

### 3. Два режима выполнения SQL (как виртуальные таблицы 1С)

```
💡 ERP Insight: Виртуальные таблицы регистра
┌─────────────────────────────────────────────────────
│ 1С: Платформа сама выбирает: читать из таблицы итогов
│     (быстро) или пересчитывать из движений (точно).
│     Разработчик пишет: РегистрНакопления.Остатки(&Период)
│     и не думает о реализации.
│ ─────────────────────────────────────────────────────
│ Metapus: ReportExecutor выбирает стратегию автоматически:
│   - asOfDate == nil или today → читать из reg_stock_balances (O(1))
│   - asOfDate в прошлом → CTE по reg_stock_movements
│ Разработчик отчёта не знает о деталях реализации.
└─────────────────────────────────────────────────────
```

```go
// internal/domain/reports/executor/stock_balance.go

func (e *StockBalanceExecutor) Execute(ctx, filter) (*Result, error) {
    if isCurrentDate(filter.AsOfDate) {
        return e.fromBalancesTable(ctx, filter)  // O(1) per row
    }
    return e.fromMovementsCTE(ctx, filter)        // historical
}
```

### Generic хук `useReportPage`

```typescript
// frontend/hooks/useReportPage.ts

function useReportPage<F, R>(reportKey: string) {
    const meta = useReportMeta(reportKey)   // GET /metadata/reports/{key}
    const [filter, setFilter] = useReportFilter<F>(reportKey)  // URL state
    const [result, setResult] = useState<R | null>(null)
    const [status, setStatus] = useState<"idle"|"loading"|"done"|"error">()

    const generate = useCallback(async () => {
        setStatus("loading")
        const data = await api.reports.execute<F, R>(reportKey, filter)
        setResult(data)
        setStatus("done")
    }, [filter])

    return { meta, filter, setFilter, result, status, generate }
}
```

## Группировка и итоги — архитектура

```
Backend: плоский массив, отсортированный по полям группировки
         (ORDER BY warehouse_name, product_name)
         ↓
Frontend: buildDisplayRows(items, activeGroupBy) → DisplayRow[]
         ↓
ReportTable: рендерит group-header строки + data строки + sticky footer
```

```typescript
type DisplayRow =
  | { kind: "group";  depth: number; label: string; subtotals: Record<string, number> }
  | { kind: "data";   depth: number; item: ReportItem }
  | { kind: "footer"; totals: Record<string, number> }
```

Группировка — **чисто фронтовая трансформация** плоского массива. Backend не знает о группировке — только сортирует. Это позволяет менять группировку без нового запроса к серверу.

## Что это даёт по сравнению с текущей реализацией

| Аспект | Сейчас | Идеально |
|---|---|---|
| Новый отчёт | 4 файла (types, service, repo, handler) | 1 struct с `Meta()` + `Execute()` |
| RLS | Не применяется | Автоматически через `ScopeDimensions` |
| Экспорт | Нет | Streaming CSV/XLSX из коробки |
| Метаданные для UI | Нет | `GET /metadata/reports/{key}` |
| Группировка | Не реализована | Фронт трансформирует плоский массив |
| Fast path | Нет (всегда CTE) | `reg_stock_balances` для текущей даты |
| Варианты настроек | Нет | `api.preferences` + именованные варианты |
| Новый отчёт в Extension API | Невозможно | `reg.RegisterReport(&MyReport{})` |

Текущий `RouteRegistration` остаётся как fallback для нестандартных случаев. [4](#6-3) 
Новый `ReportRegistration[F, R]` — для всех аналитических отчётов. [5](#6-4)

## 10. Итоговая карта компонентов

```
ReportPage (новый компонент)
├── FormToolbar                    ← существующий компонент
│   ├── primaryAction="Сформировать"
│   ├── extraMenuItems=[Экспорт CSV, Экспорт XLSX, Печать]
│   └── toolbarIcons=[Настройки, Копировать ссылку]
│
├── FilterSidebar                  ← существующий компонент
│   ├── fieldsMeta=report.meta.filters   ← из GET /metadata/reports/{key}
│   ├── periodField="asOfDate"           ← если отчёт на дату
│   ├── detailsContent=<DrillDownPanel>  ← новый компонент
│   └── onFilterValuesChange → setDraftFilter
│
└── ReportResultArea (новый компонент)
    ├── ReportDataToolbar          ← новый компонент (расширение DataToolbar)
    │   ├── ColumnChooserPopover   ← существующий компонент
    │   ├── GroupBySelector        ← новый компонент (ToggleGroup)
    │   └── GranularitySelector    ← новый компонент (только для оборотов)
    │
    ├── ScrollArea                 ← существующий компонент
    │   └── ReportTable            ← новый компонент (расширение DataTable)
    │       ├── <tr class="group-header"> для group rows
    │       └── <tr class="data-row">    для data rows
    │
    └── ReportTotalsFooter         ← новый компонент (расширение DocumentTotalsFooter)
        └── несколько числовых итогов (не только сумма)
```

`DocumentTotalsFooter` сейчас заточен под `totalAmount + totalVat`. Для отчётов нужен `ReportTotalsFooter` с произвольным набором итоговых полей из `ReportMeta.totals`. [4](#8-3) 

`DataTable` сейчас требует `T extends { id: string }` и не поддерживает group-header строки. `ReportTable` — отдельный компонент, принимающий `DisplayRow[]` вместо `T[]`, чтобы не ломать существующий `DataTable`. [5](#8-4)
