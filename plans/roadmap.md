## Roadmap

### Фаза 0: Техническая стабилизация (1-2 недели)

**Цель:** Закрыть технический долг, не добавляя бизнес-функциональности.

| # | Задача | Тип | Обоснование |
|---|--------|-----|-------------|
| 0.1 | CI/CD pipeline (GitHub Actions: lint, test, build) | Infra | Без CI невозможна стабильная разработка. Каждый следующий спринт будет генерировать регрессии |
| 0.2 | Тесты для PostingEngine, Auth, CatalogService | Test | Критический путь должен быть покрыт до расширения функциональности |
| 0.3 | Выделить `BaseDocumentService[T]` | Refactor | GoodsReceipt и GoodsIssue services дублируют логику Create/Update/Delete/Post/Unpost. Каждый новый документ будет копировать этот boilerplate |
| 0.4 | Исправить `Dockerfile` (Go 1.22 → 1.24) | Bugfix | Несовместимость с go.mod |
| 0.5 | Убрать JWT secret fallback, добавить `.env` в `.gitignore` | Security | Потенциальная дыра в production |
| 0.6 | Frontend: недостающие страницы (GoodsIssue, Units, Currencies, Contracts, VatRates) | Frontend | Каталоги и документы существуют на backend, но недоступны через UI |
| 0.7 | Frontend: страницы отчётов (Stock Balance, Stock Turnover, Document Journal) | Frontend | API есть, UI нет |

**Результат:** Стабильная платформа с CI, тестами и полным UI для существующего функционала.

### Фаза 1: Суммовой учёт — Себестоимость и Взаиморасчёты (2-3 недели)

**Цель:** Перейти от чисто количественного учёта к суммовому. Без этого невозможны ни финансовые отчёты, ни расчёт прибыли.

| # | Задача | Тип | Детали |
|---|--------|-----|--------|
| 1.1 | Регистр накопления **Себестоимость товаров** | Register | Вид: Остатки. Измерения: Склад, Номенклатура. Ресурсы: Количество, Сумма. **Рекомендация эксперта: "По средней"** — для MVP это оптимально. Можно совместить с reg_stock, добавив ресурс `amount` |
| 1.2 | Регистр накопления **Взаиморасчёты с контрагентами** | Register | Вид: Остатки. Измерения: Контрагент, Договор. Ресурс: Сумма. Необходим для контроля долгов |
| 1.3 | Расширить `MovementSet` в Posting Engine | Backend | Добавить `CostMovements []entity.CostMovement` и `SettlementMovements []entity.SettlementMovement` (сейчас только StockMovements, остальные удалены по YAGNI) |
| 1.4 | Доработать **Поступление товаров** | Backend | Добавить движения по себестоимости (приход) и взаиморасчётам (увеличение долга поставщику) |
| 1.5 | Доработать **Расход товаров** → переименовать в **Реализацию** | Backend | Добавить движения: себестоимость (расход), взаиморасчёты (увеличение долга покупателя). Расчёт себестоимости списания по средней |
| 1.6 | Регистр накопления **Продажи** (обороты) | Register | Вид: Обороты. Измерения: Номенклатура, Контрагент. Ресурсы: Количество, Выручка, Себестоимость. Для отчёта о прибыли |
| 1.7 | Отчёт **Продажи и Валовая прибыль** | Report | Выручка - Себестоимость = Валовая прибыль. По номенклатуре, контрагентам, периодам |
| 1.8 | Отчёт **Взаиморасчёты** (ведомость) | Report | Кто должен нам, кому должны мы. По контрагентам и договорам |

**Результат:** Полноценный суммовой учёт. Владелец видит прибыль и долги.

> **Решение по архитектуре:** Себестоимость рекомендую объединить с reg_stock — добавить ресурс `amount BIGINT` в movements и balances. Это проще, чем отдельный регистр, и достаточно для "по средней". При переходе на ФИФО потребуется отдельный регистр партий.

### Фаза 2: Заказы — ключевой бизнес-процесс автозапчастей (3 недели)

**Цель:** Реализовать цепочку Заказ покупателя → Заказ поставщику → Поступление → Реализация. Это **основной бизнес-процесс** магазина автозапчастей.

| # | Задача | Тип | Детали |
|---|--------|-----|--------|
| 2.1 | Документ **Заказ покупателя** | Document | Шапка: покупатель, договор, дата отгрузки. Табличная часть: номенклатура, количество, цена. Статусы: Новый → В работе → Готов к выдаче → Закрыт |
| 2.2 | Регистр накопления **Заказы покупателей** | Register | Вид: Остатки. Измерения: Заказ, Номенклатура. Ресурс: Количество (заказано), Обеспечено. Контролирует, что привезли под клиента |
| 2.3 | Документ **Заказ поставщику** | Document | Формируется на основании потребностей (Заказов покупателей). Ссылка на исходный заказ покупателя в строках |
| 2.4 | Механизм **Ввод на основании** | Backend | Generic-механизм создания документа на основании другого: Заказ покупателя → Заказ поставщику → Поступление → Реализация. API: `POST /document/{type}/{id}/create-based` |
| 2.5 | Отчёт **Обеспечение заказов** | Report | Менеджер видит: Клиент X заказал деталь Y, она приехала на склад → пора звонить |
| 2.6 | Связь Поступление → Заказ | Backend | При оприходовании товара по заказу — автоматическое обеспечение строк заказа покупателя |
| 2.7 | Frontend: Заказ покупателя (список + форма) | Frontend | С отображением статуса обеспечения в каждой строке |
| 2.8 | Frontend: Заказ поставщику (список + форма) | Frontend | С кнопкой "Создать на основании заказов покупателей" |

**Результат:** Полный цикл "клиент попросил → заказали у поставщика → товар приехал → клиент уведомлён → товар продан".

### Фаза 3: Ценообразование (1-2 недели)

**Цель:** Управление ценами: закупочные, розничные, оптовые.

| # | Задача | Тип | Детали |
|---|--------|-----|--------|
| 3.1 | Справочник **Виды цен** | Catalog | Закупочная, Розничная, Оптовая (расширяемый) |
| 3.2 | Регистр сведений **Цены номенклатуры** | Register | Периодический. Измерения: Номенклатура, Вид цены. Ресурс: Цена. Хранит историю изменения цен |
| 3.3 | Документ **Установка цен** | Document | Массовое задание цен для списка номенклатуры. Делает движения в регистре цен |
| 3.4 | Автоматическое заполнение цен | Backend | При создании Заказа/Реализации — подтягивание актуальной цены из регистра по виду цены контрагента |
| 3.5 | Отчёт **Прайс-лист** | Report | Актуальные цены + остатки для быстрой консультации менеджера |

**Результат:** Централизованное управление ценами вместо ручного ввода в каждом документе.

### Фаза 4: Денежные средства (2 недели)

**Цель:** Учёт наличных и безналичных денег.

| # | Задача | Тип | Детали |
|---|--------|-----|--------|
| 4.1 | Справочник **Кассы** | Catalog | Касса организации, Мелкая касса и т.д. |
| 4.2 | Справочник **Банковские счета** | Catalog | Расчётные счета организации |
| 4.3 | Регистр накопления **Денежные средства** | Register | Вид: Остатки. Измерения: Касса/Счёт. Ресурс: Сумма |
| 4.4 | Документ **ПКО** (Приходный кассовый ордер) | Document | Поступление наличных. Движения: ДС (+), Взаиморасчёты (-) |
| 4.5 | Документ **РКО** (Расходный кассовый ордер) | Document | Выдача наличных. Движения: ДС (-), Взаиморасчёты (+) |
| 4.6 | Документы **Поступление/Списание по р/с** | Document | Безнал. Аналогично ПКО/РКО, но для банковского счёта |

**Результат:** Владелец видит сколько денег есть, сколько пришло, сколько ушло.

### Фаза 5: Складские операции (1-2 недели)

**Цель:** Контроль фактических остатков, инвентаризация.

| # | Задача | Тип | Детали |
|---|--------|-----|--------|
| 5.1 | Документ **Инвентаризация товаров** | Document | Сличение учётных и фактических остатков. Не делает движений — это "акт сверки" |
| 5.2 | Документ **Оприходование излишков** | Document | На основании инвентаризации. Движения: остатки (+), себестоимость (+) |
| 5.3 | Документ **Списание недостач** | Document | На основании инвентаризации. Движения: остатки (-), себестоимость (-) |
| 5.4 | Перемещение товаров (опционально) | Document | Между складами. Движения: расход с одного, приход на другой |

**Результат:** Магазин может проводить инвентаризацию и корректировать остатки.

### Фаза 6: Автозапчастевая специфика + АРМ (2-3 недели)

**Цель:** Киллер-фичи, которые выделят продукт среди конкурентов.

| # | Задача | Тип | Детали |
|---|--------|-----|--------|
| 6.1 | Справочник **Производители (Бренды)** | Catalog | Отдельный от Контрагентов. Номенклатура ссылается на бренд. Уникальность: Артикул + Бренд |
| 6.2 | Регистр сведений **Аналоги номенклатуры** | Register | Непериодический. Измерения: Номенклатура, Аналог. Позволяет менеджеру предлагать замены |
| 6.3 | **АРМ менеджера** (Рабочее место) | Frontend | Одна форма: поиск (артикул/штрихкод/наименование) → остатки + цены + **аналоги** → "В корзину" → "Оформить заказ/продажу". Keyboard-first, оптимизирован для скорости |
| 6.4 | Расширенный поиск номенклатуры | Backend | Поиск по артикулу с автоматической проверкой аналогов. Если нет в наличии — показать аналоги, которые есть |
| 6.5 | Регистр сведений **Штрихкоды** (опционально) | Register | Если нужна поддержка нескольких штрихкодов на одну номенклатуру. Для MVP достаточно поля в cat_nomenclature |
| 6.6 | Quick-view карточка товара | Frontend | Всплывающая карточка: фото, артикул, бренд, аналоги, остатки по складам, цены по видам цен |

**Результат:** Менеджер обрабатывает клиента за 30 секунд вместо 3 минут. Аналоги продают то, что есть в наличии.

## Рекомендуемый порядок для MVP

Минимальный MVP для запуска реального магазина автозапчастей:

| Приоритет | Фаза | Недели | Результат |
|-----------|-------|--------|-----------|
| **P0** | Фаза 0: Стабилизация | 1-2 | CI, тесты, полный UI |
| **P0** | Фаза 1: Себестоимость + Взаиморасчёты | 2-3 | Суммовой учёт, прибыль |
| **P0** | Фаза 2: Заказы | 3 | Основной бизнес-процесс |
| **P1** | Фаза 3: Ценообразование | 1-2 | Управление ценами |
| **P1** | Фаза 4: Денежные средства | 2 | Касса и банк |
| **P1** | Фаза 6: АРМ + Аналоги | 2-3 | Киллер-фича для продавцов |
| **P2** | Фаза 5: Склад | 1-2 | Инвентаризация |
| **P2** | Фаза 8: Quality | 2 | Production-ready |
| | | **~16-19** | **Полный MVP** |

**Минимальный жизнеспособный срез (MVP-0):** Фазы 0 + 1 + 2 = **~7-8 недель**. Этого достаточно для запуска магазина с ограниченным функционалом: принимать заказы, закупать, продавать, видеть прибыль.

## Архитектурные рекомендации

### 1. Себестоимость: "По средней" vs ФИФО

Для MVP — **по средней** (рекомендация эксперта). Реализация: при расходе товара средняя себестоимость = `SUM(amount) / SUM(quantity)` из регистра остатков. Не требует партионного учёта.

**Как расширить до ФИФО позже:** Добавить регистр партий (`reg_cost_layers`) с измерениями: Склад, Номенклатура, Документ поступления. При расходе — списание по дате поступления (от старых к новым).

### 2. Объединение reg_stock + Себестоимость

Рекомендую добавить `amount BIGINT` в `reg_stock_movements` и `reg_stock_balances` вместо создания отдельного регистра. Для MVP "по средней" этого достаточно: одна таблица, одна запись на Склад+Товар.

### 3. BaseDocumentService[T]

По аналогии с `CatalogService[T]` — выделить generic-сервис для документов:
- `Create`, `Update`, `Delete`, `GetByID`, `List` — стандартные операции
- `Post`, `Unpost` — через PostingEngine
- `SetDeletionMark` — с проверкой `CanModify()`
- Hooks: `BeforePost`, `AfterPost`, `BeforeCreate`, etc.

Это сократит boilerplate при добавлении каждого нового документа (Заказ покупателя, Заказ поставщику, ПКО, РКО и т.д.).

### 4. Ввод на основании (Document Chains)

Generic-механизм:
```
type DocumentBasis interface {
    GetBasisType() string
    GetBasisID() id.ID
    FillFromBasis(ctx context.Context, basis interface{}) error
}
```

Конфигурация допустимых цепочек в Metadata Registry:
```
ЗаказПокупателя → ЗаказПоставщику
ЗаказПоставщику → Поступление
ЗаказПокупателя → Реализация
Поступление → РКО (оплата поставщику)
Реализация → ПКО (оплата от покупателя)
```

### 5. Регистры сведений (Info Registers)

Текущая архитектура поддерживает только регистры накопления (accumulation). Для Цен, Аналогов, Штрихкодов нужен отдельный паттерн — **Регистр сведений**:
- Не связан с документами-регистраторами (или связан опционально)
- Может быть периодическим (цены — с датой актуальности) или непериодическим (аналоги)
- Запрос: "Дай актуальное значение на дату X"

Рекомендую создать `BaseInfoRegister[T]` с generic-реализацией `GetSliceLast(date, dimensions)`.
