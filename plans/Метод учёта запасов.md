## 💡 ERP Insight: Метод учёта запасов

```
💡 ERP Insight: Inventory Valuation Method (FIFO / Average)
┌─────────────────────────────────────────────────────
│ 1С:      Учётная политика организации (периодический
│          регистр сведений). Метод: ФИФО или Средняя.
│          При расходе — партионный учёт (регистр
│          "Партии товаров на складах") или расчёт
│          средней по остаткам. Смена метода — только
│          с начала периода.
│
│ ERPNext: Item Valuation Method на уровне Item или
│          глобально в Stock Settings. FIFO = Stock
│          Ledger Entry с incoming_rate. Average =
│          valuation_rate в Bin. Repost Item Valuation
│          при ретроактивных изменениях.
│
│ SAP:     Material Ledger (ML) + Actual Costing.
│          Метод задаётся в Material Master (Costing 1).
│          Moving Average Price (MAP) или Standard Price.
│          FIFO через Batch Management + MBGF.
│
│ Odoo:    Costing Method на уровне Product Category
│          (standard / average_price / fifo).
│          FIFO через stock.valuation.layer (SVL) —
│          отдельная таблица слоёв стоимости.
│          Смена метода — wizard с пересчётом.
│ ─────────────────────────────────────────────────────
│ Лучшее:  Odoo 17 — SVL (Stock Valuation Layer) как
│          отдельный иммутабельный регистр слоёв.
│          Чисто, аудируемо, не смешивает qty и cost.
│
│ Metapus: Привязка к Организации — правильно.
│          Реализация через существующий CostRecorder
│          + новый регистр cost_layers для ФИФО.
│          Средняя — через reg_cost_balances (уже есть).
│ Почему:  Организация = юридическое лицо = субъект
│          учётной политики. Один тенант может иметь
│          несколько организаций с разными методами.
```

## Архитектурное решение

### Где хранится настройка

**Правильно: в `cat_organizations`** — уже реализовано. Это аналог "Учётной политики организации" в 1С.

```
cat_organizations.inventory_method = 'fifo' | 'average'
```

**Не в `sys_settings`** — там системные настройки (нумерация, производительность), не бизнес-политика. [5](#0-4) 

---

### Два метода — два алгоритма в `CostRecorder`

```
┌─────────────────────────────────────────────────────────────────┐
│  Приход (GoodsReceipt)                                          │
│  GenerateCostMovements() → CostMovement{qty, amount}            │
│  Одинаково для ФИФО и Средней — просто записываем партию        │
│                                                                 │
│  Расход (GoodsIssue)                                            │
│  ┌──────────────────────┬──────────────────────────────────┐    │
│  │ AVERAGE              │ FIFO                             │    │
│  │                      │                                  │    │
│  │ avg = amount / qty   │ Читаем reg_cost_layers           │    │
│  │ из reg_cost_balances │ ORDER BY period ASC              │    │
│  │                      │ Списываем слои по очереди        │    │
│  │ expense_amount =     │ expense_amount = сумма слоёв     │    │
│  │   expense_qty * avg  │ (частичное списание последнего)  │    │
│  └──────────────────────┴──────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────────┘
```

### Схема данных

**Для Средней стоимости** — достаточно существующего `reg_cost_balances`:

```sql
-- Уже есть: reg_cost_balances
-- warehouse_id, product_id, currency_id → quantity, amount
-- Средняя цена = amount / quantity (в момент расхода)
```

**Для ФИФО** — нужен новый регистр слоёв (аналог Odoo `stock.valuation.layer`):

```sql
-- Новая таблица: reg_cost_layers
CREATE TABLE reg_cost_layers (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    recorder_id     UUID NOT NULL,          -- документ-источник
    recorder_type   TEXT NOT NULL,
    period          TIMESTAMPTZ NOT NULL,   -- дата прихода (для FIFO-порядка)
    warehouse_id    UUID NOT NULL,
    product_id      UUID NOT NULL,
    currency_id     UUID NOT NULL,
    qty_in          BIGINT NOT NULL,        -- пришло (×10000)
    qty_remaining   BIGINT NOT NULL,        -- остаток в слое (уменьшается при расходе)
    unit_cost       BIGINT NOT NULL,        -- стоимость единицы (MinorUnits)
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);
-- Индекс для FIFO-выборки:
CREATE INDEX ON reg_cost_layers (warehouse_id, product_id, period ASC)
    WHERE qty_remaining > 0;
```

### Поток данных при проведении

```
GoodsIssue.GenerateCostMovements(ctx)
    │
    ├── 1. Получить organizationID из документа
    ├── 2. Загрузить organization.InventoryMethod
    │
    ├── [average] ──────────────────────────────────────────────
    │   ├── GetCostBalance(warehouseID, productID, currencyID)
    │   │   → {quantity: 100, amount: 50000}
    │   ├── avgUnitCost = amount / quantity = 500
    │   └── CostMovement{expense, qty=10, amount=5000}
    │
    └── [fifo] ─────────────────────────────────────────────────
        ├── GetFIFOLayers(warehouseID, productID, ORDER BY period)
        │   → [{qty_remaining:30, unit_cost:480}, {qty_remaining:80, unit_cost:510}]
        ├── Списываем 10 единиц из первого слоя
        │   → expense_amount = 10 * 480 = 4800
        │   → UPDATE reg_cost_layers SET qty_remaining = 20
        └── CostMovement{expense, qty=10, amount=4800}
```

### Где читать `inventoryMethod` в Go

Метод нужно читать **в момент проведения**, внутри транзакции. Есть два варианта:

**Вариант A: В `GenerateCostMovements()` документа** — документ сам знает свою организацию и запрашивает метод через `OrgRepo`:

```go
// internal/content/documents/goods_issue/model.go
func (g *GoodsIssue) GenerateCostMovements(ctx context.Context) ([]entity.CostMovement, error) {
    org, err := g.orgRepo.GetByID(ctx, g.OrganizationID)
    if err != nil {
        return nil, err
    }
    switch org.InventoryMethod {
    case "average":
        return g.generateAverageCostMovements(ctx)
    case "fifo":
        return g.generateFIFOCostMovements(ctx)
    }
    return nil, fmt.Errorf("unknown inventory method: %s", org.InventoryMethod)
}
```

**Вариант B: В `CostRecorder.RecordFromSet()`** — рекордер читает метод и применяет алгоритм. Чище с точки зрения разделения ответственности, но рекордер становится "умнее".

**Рекомендация: Вариант A** — документ реализует `CostMovementSource`, и именно он знает контекст (организацию). Это соответствует паттерну Visitor в Metapus: документ генерирует движения, рекордер только записывает.

## ⚠️ ERP Anti-Pattern: Смена метода задним числом

```
⚠️ ERP Anti-Pattern: Ретроактивная смена метода учёта
┌─────────────────────────────────────────────────────
│ Проблема: В ERPNext смена Valuation Method на Item
│           требует "Repost Item Valuation" — пересчёт
│           всей истории движений. На больших базах
│           это часы работы и блокировки.
│ Система:  ERPNext
│ Причина:  Стоимость хранится в движениях, а не в
│           слоях. Пересчёт = UPDATE на тысячах строк.
│ Metapus:  Запретить смену метода, если есть
│           проведённые документы в текущем периоде.
│           Смена — только с начала нового периода
│           (как в 1С: учётная политика на год).
│           Валидация в Organization.Validate():
│           if hasPostedDocuments(period) → error
└─────────────────────────────────────────────────────
```

## Итоговая карта реализации

| Слой | Что делать |
|------|-----------|
| **БД** | Добавить `reg_cost_layers` для ФИФО; `reg_cost_balances` уже есть для Средней |
| **`entity/register.go`** | Добавить `CostLayer` struct |
| **`cat_organizations`** | Поле `inventory_method` уже есть в БД и UI |
| **`GoodsReceipt.GenerateCostMovements()`** | Создавать `CostMovement{receipt}` + `CostLayer` при ФИФО |
| **`GoodsIssue.GenerateCostMovements()`** | Читать `org.InventoryMethod`, применять алгоритм |
| **`CostRecorder`** | При ФИФО: обновлять `qty_remaining` в слоях внутри транзакции |
| **`Organization.Validate()`** | Запрет смены метода при наличии проведённых документов |
| **Отчёт** | `reg_cost_balances` → средняя цена; `reg_cost_layers` → ФИФО-стоимость |

Ключевой момент: `reg_cost_balances` уже хранит `quantity + amount` — для **Средней** этого достаточно прямо сейчас. **ФИФО** требует дополнительного регистра слоёв, без которого невозможно знать, какая партия списывается. [6](#0-5) [7](#0-6)
