# UX-аудит интерфейса настроек безопасности `/settings`

> Детальный анализ интерфейса управления правами доступа в Metapus ERP.
> Исследованы разделы: **Пользователи и роли**, **Профили безопасности** (RLS / FLS / CEL-правила).

## 1. Анализ текущих проблем

### 🔴 1.1 RLS — UUID вместо человеческих имён

![RLS вкладка профиля «Менеджер»](C:\Users\user\.gemini\antigravity\brain\4b566cbe-294b-407e-83e2-0d3a051925f9\security_profile_manager_rls_config_1773283922016.png)

**Проблема:** В RLS-блоке администратор видит поле «UUID через запятую» вида `019caf03-5dc2-7c16-9269-ed55f4abcd18`. Администратор **не может** по UUID понять, какая именно организация, склад или контрагент имеется в виду. Это делает настройку крайне подверженной ошибкам (копипаст UUID, опечатки, не тот UUID).

**Когнитивная нагрузка:** Запредельная. Нарушены все принципы: закон Хика (невозможно осознанно выбрать из UUID), принцип ясности.

---

### 🔴 1.2 FLS — DSL без подсказок и визуального конструктора

![FLS вкладка профиля «Менеджер»](C:\Users\user\.gemini\antigravity\brain\4b566cbe-294b-407e-83e2-0d3a051925f9\security_profile_manager_fls_details_1773283944633.png)

**Проблема:** Настройка Field-Level Security требует от администратора знания DSL-синтаксиса (`*, -unit_price`). При этом нет:
- Списка доступных полей сущности
- Автодополнения
- Валидации ввода (можно ввести несуществующее поле)
- Визуального переключателя «Скрыть/Показать поле»

**Рисковый сценарий:** Администратор может вписать `*,-uint_price` (опечатка) — ограничение не сработает, и никакой ошибки не будет показано.

### 🔴 1.3 Действия над пользователями — stub-функции

![Меню действий пользователя](C:\Users\user\.gemini\antigravity\brain\4b566cbe-294b-407e-83e2-0d3a051925f9\settings_user_actions_menu_1773283824101.png)

**Проблема:** Кнопки «Редактировать», «Сменить роль», «Заблокировать» в dropdown-меню пользователя — **заглушки без обработчиков** (в коде `<DropdownMenuItem>Редактировать</DropdownMenuItem>` без `onClick`). Нажатие ничего не делает. Это вводит администратора в заблуждение: кнопки выглядят кликабельными, но не работают.

### 🟡 1.4 Нет связи «Пользователь ↔ Профиль безопасности»

**Проблема:** В таблице пользователей показаны: ФИО, email, роли, статус. Но **не показан назначенный Security Profile**. Это критически важная информация — именно профиль определяет RLS/FLS/CEL-ограничения. Администратору приходится заходить в каждый профиль, чтобы понять, кто к нему привязан.

Аналогично: в карточке профиля безопасности **нет списка привязанных пользователей**.

## 2. Скрытые проблемы и упущенные сценарии

### 2.1 Цепочка CEL-правил не видна в контексте

На вкладке «Правила» видно количество правил и их содержимое, но:
- **Нет приоритетов** — не видно поля `priority` при просмотре правила (оно не отображается)
- **Нет информации о порядке применения** — администратор не понимает, какое правило сработает первым
- **Нет сценария «что если»** — нельзя протестировать правило на конкретном документе/пользователе

### 2.2 Нет контроля дубликатов FLS-политик

Можно создать две FLS-политики для одной сущности + действия (например, 2×`goods_receipt:read`). Поведение при конфликте неясно, UI не предупреждает.

### 2.3 Нет роли-помощника для нового профиля

Создание профиля с нуля — экспертная задача. Нужно помнить:
- Какие сущности есть в системе
- Какие у них поля
- Синтаксис DSL для FLS
- Синтаксис CEL для правил

Нет шаблонов (presets) и мастера (wizard).

### 2.4 Нет визуализации «эффективных прав»

После настройки RLS + FLS + CEL + RBAC-ролей администратор не может увидеть **итоговую картину**: «Что конкретно видит/может Алексей Петров?». Нужен экран «Эффективные права пользователя».

### 2.5 Нет аудита изменений профиля

Кто и когда изменил профиль безопасности? Это критично для security compliance, но в интерфейсе нет даты последнего изменения и лога правок.

### 2.6 Роли — read-only без контекста

![Вкладка ролей](C:\Users\user\.gemini\antigravity\brain\4b566cbe-294b-407e-83e2-0d3a051925f9\settings_roles_list_1773283810281.png)

Карточки ролей показывают название и описание, но:
- **Не указано количество пользователей** с этой ролью
- **Не перечислены разрешения** (permissions) роли — администратор не видит, что конкретно даёт роль `Manager`
- **Нет действий**: нельзя ни редактировать, ни просмотреть детали роли

### 2.7 Отсутствует кнопка «Создать пользователя»

В разделе «Пользователи» нет кнопки для создания нового пользователя. Workflow администратора: «создать аккаунт → назначить роль → привязать профиль безопасности» — разрывается на первом шаге.

## 4. Конкретный UI-рефакторинг

### 4.1 RLS: Picker вместо UUID-ввода

**Сейчас:** `<Input placeholder="UUID через запятую" className="font-mono" />`

**Нужно:** Multi-select / Combobox с поиском по названию. При добавлении организации — поиск по имени из API (`/api/v1/catalogs/organizations`), отображение бейджей с именами. Под капотом хранятся UUID, но пользователь видит только названия.

```
┌─ Организации ──────────────────────────────┐
│  [× ООО «Альфа»]  [× ООО «Бета»]         │
│  [🔍 Поищите организацию...]                │
└─────────────────────────────────────────────┘
```

### 4.2 FLS: Визуальный конструктор полей

**Сейчас:** Текстовый input `*, -unit_price`

**Нужно:** Для каждой сущности загружать список полей из метаданных. Для каждого поля — чекбокс «Видно»/«Скрыто».

```
┌─ Поступление товаров — Чтение ──────────────╮
│ Шапка:                                       │
│  ☑ number  ☑ date  ☑ supplier  ☐ unit_price │
│  ☑ quantity  ☐ total_amount  ☐ total_vat    │
│ Табличная часть (lines):                     │
│  ☑ product  ☑ quantity  ☐ unit_price        │
╰──────────────────────────────────────────────╯
```

Для опытных — toggle «Режим DSL» с текущим input.

### 4.3 Пользователи: Колонка «Профиль безопасности»

В таблицу пользователей добавить колонку после «Роли»:

| Пользователь | Роли | Профиль безопасности | Статус |
|---|---|---|---|
| Алексей Петров | `Manager` | `Менеджер` | Активен |
| Елена Смирнова | [User](file:///c:/Users/user/go/src/metapus/frontend/components/settings/users-roles-section.tsx#87-223) | `Viewer` | Активен |

### 4.4 Профиль: Вкладка «Пользователи»

В Sheet профиля безопасности добавить 5-ю вкладку **«Пользователи»** со списком привязанных пользователей + кнопкой «Привязать пользователя».

### 4.5 Роли: Раскрытие permissions

Сделать карточку роли кликабельной → открытие drawer/sheet со списком всех permissions этой роли, сгруппированных по entity_type:

```
Accountant
├── catalog:warehouse:read, create, update
├── catalog:counterparty:read, create
├── document:goods_receipt:read, create, update, post
└── register:stock:read
```

### 4.6 CEL-правила: Приоритет и описание

В карточке правила добавить:
- Поле **Приоритет** с визуальным индикатором (число)
- Drag&drop для упорядочивания приоритетов
- Поле **Описание** для пояснения, зачем правило нужно
- Модификация формы создания: добавить поле `priority` (сейчас оно не показано при создании)

### 4.7 Кнопки действий пользователей — подключить обработчики

Реализовать (или убрать до реализации):
- **Редактировать** → Sheet с полями: имя, фамилия, email, статус, admin-флаг
- **Сменить роль** → Dialog с multi-select ролей
- **Заблокировать** → Confirm dialog + вызов API деактивации
- **Назначить профиль безопасности** → dialog select

### 4.8 CEL — «Песочница» для тестирования

Кнопка «Проверить синтаксис» уже работает по `onBlur`, но нужно расширить:
- Подсказка доступных переменных (`doc.posted`, `doc.total_amount`, `user.isAdmin`, `now`)
- Мини-справочник примеров прямо в UI (collapsible)
- Кнопка «Тест на документе» — выбрать реальный документ и пользователя → показать, какие правила сработают

## 5. Улучшение связанного функционала

### 5.1 Экран «Эффективные права пользователя»

Новая вкладка «Доступ» в карточке пользователя или отдельная страница. Показывает:
- Результирующие RBAC-разрешения (из ролей)
- Наложенные RLS-фильтры (из Security Profile)
- Ограничения FLS (какие поля скрыты по сущностям)
- Активные CEL-правила
- Организации, к которым есть доступ

### 5.2 Кнопка «Создать пользователя»

В тулбаре раздела «Пользователи» добавить кнопку `+ Создать пользователя` → форма с полями: email, имя, фамилия, пароль, роли, профиль безопасности, организации.

### 5.3 Шаблоны профилей безопасности (presets)

При создании нового профиля — предложить выбрать из типичных шаблонов:
- **Только просмотр** — read-only, скрытые цены
- **Менеджер** — CRUD без проведения, ограниченные организации
- **Бухгалтер** — полный доступ к документам, ограниченные склады
- **Кладовщик** — только складские операции

### 5.4 Аудит-лог изменений Security

В карточке профиля добавить вкладку «История» с записями:
- `12.03.2026 10:32 — admin@metapus.io изменил RLS: добавлена организация «ООО Бета»`

### 5.5 Валидация при сохранении профиля

Перед сохранением проверять:
- Нет ли дублирующихся FLS-политик (одна сущность+действие)
- Не указаны ли несуществующие UUID в RLS
- Корректность всех CEL-выражений

## 7. Психологическое обоснование

| Изменение | Принцип |
|-----------|---------|
| **Picker вместо UUID** | **Признавайте, а не вспоминайте** (Якоб Нильсен). Человек не может запомнить UUID; выбор из списка снижает ошибки на порядок |
| **Визуальный конструктор FLS** | **Закон Хика** — чекбоксы быстрее осваиваются, чем DSL-синтаксис. **Предотвращение ошибок** — невозможно вписать несуществующее поле |
| **Человечные названия вкладок** | **Принцип ясности** (Стив Круг) — «Видимость данных» понятнее, чем «RLS», широкой аудитории |
| **Колонка профиля у пользователя** | **Принцип близости** (Гештальт) — связанная информация должна быть рядом, а не разбросана по разным экранам |
| **Эффективные права** | **Ментальная модель** (Дон Норман) — администратор думает «что может этот пользователь?», а не «как сложить RBAC + RLS + FLS + CEL» |
| **Шаблоны профилей** | **Прогрессивное раскрытие** — начать с готового шаблона, затем кастомизировать. Снижает порог входа |
| **Аудит-лог** | **Обратная связь** — система должна сообщать, что было сделано и кем. Критично для security |

## 8. Если бы это был мой продукт

### 🏆 Улучшение №1: «Эффективные права пользователя»
Главный экран, который покажет **итоговый результат** всех настроек безопасности для конкретного пользователя. Без него администратор настраивает вслепую — он не видит конечный эффект наложения ролей + профиля + CEL-правил. Это снижает доверие к системе безопасности и увеличивает риск ошибок конфигурации.

### 🏆 Улучшение №2: Замена UUID-инпутов на Picker с поиском
Это единственное изменение, которое **мгновенно** превратит RLS-настройку из «экспертного инструмента для разработчиков» в «понятный бизнес-инструмент для администратора». ROI максимальный при минимальных усилиях — нужен только Combobox с запросом к уже существующему API каталогов.
